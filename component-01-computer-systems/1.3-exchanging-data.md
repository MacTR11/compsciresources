## 1.3.1 Compression, encryption and hashing

### Compression

**Compression** means reducing the size of a file so it uses less storage space and can be transmitted faster (using less bandwidth).

**Lossless compression** reduces file size without permanently removing any data. The original file can be reconstructed **exactly** when decompressed. It works by recording patterns/rules that describe the data more efficiently. Used where you cannot afford to lose detail (e.g. text files, executable programs, spreadsheets).

**Lossy compression** reduces file size by **permanently removing some data** (usually data the human eye/ear is less sensitive to). The original file **cannot** be reconstructed exactly. It achieves much smaller files. Used for images, audio and video where small quality loss is acceptable (e.g. JPEG, MP3).

| Feature | Lossless | Lossy |
|---|---|---|
| Data removed permanently? | No | Yes |
| Original recoverable exactly? | Yes | No |
| Typical file size reduction | Smaller reduction | Larger reduction |
| Typical uses | Text, code, ZIP, PNG | Images (JPEG), audio (MP3), video |

> 💡 **A\* extension:** Lossy is only appropriate where loss of fidelity is imperceptible or tolerable to the user. You would never apply lossy compression to a program file, a bank database or a legal document because corrupting a single byte could make it unusable or change its meaning.

### Run length encoding (RLE)

**Run length encoding** is a lossless technique that replaces **runs** (sequences of repeated identical values) with a single value and a count of how many times it repeats.

RLE is most effective when there are **long runs of repeated data** (e.g. simple images with large blocks of the same colour). It can actually *increase* file size if the data has few or no repeats, because each value now also stores a count.

### Dictionary-based compression

**Dictionary-based compression** is a lossless technique that builds a **dictionary** (an index) of recurring patterns/words in the data. Each pattern is stored once in the dictionary and replaced in the file by a short reference (e.g. an index number).

- The dictionary must be stored or sent **with** the compressed data so the file can be reconstructed.
- It is effective for text, where words and phrases repeat frequently.

> 💡 **A\* extension:** Dictionary compression trades off the overhead of storing the dictionary against the saving from replacing long/repeated patterns with short codes. For small files the dictionary overhead may outweigh the saving. (LZW is a real-world example, used in GIF and some PDF/TIFF files.)

### Encryption

**Encryption** scrambles data into an unreadable form (**ciphertext**) using an algorithm and a **key**, so that only someone with the correct key can decrypt it back to **plaintext**. It protects data **confidentiality** if it is intercepted.

**Symmetric encryption** uses the **same key** to both encrypt and decrypt the data.
- Fast and efficient.
- Problem: the key must be shared securely between sender and receiver. If the key is intercepted during exchange, the encryption is broken. This is the **key distribution problem**.

**Asymmetric encryption** (public/private key) uses a **pair of mathematically related keys**: a **public key** (shared with anyone) and a **private key** (kept secret).
- Data encrypted with the **public key** can only be decrypted with the matching **private key**.
- Solves the key distribution problem because the public key can be shared openly.
- Slower / more computationally expensive than symmetric.

> 💡 **A\* extension:** Asymmetric encryption also enables **digital signatures**. The sender encrypts a hash of the message with their **private** key; anyone can decrypt it with the sender's **public** key to verify it genuinely came from them and was not altered. This proves **authenticity** and **integrity**, not just confidentiality. In practice (e.g. HTTPS) asymmetric encryption is used to securely exchange a symmetric **session key**, then the faster symmetric encryption is used for the bulk data.

### Hashing

**Hashing** applies a **hash function** to an input to produce a fixed-length output called a **hash** (or digest). Hashing is:
- **One-way** — you cannot feasibly reverse a hash back to the original input.
- **Deterministic** — the same input always produces the same hash.

A good hash function should:
- Produce few **collisions** (different inputs giving the same hash).
- Produce a hash much smaller than the data, with a fixed length.
- Be quick to calculate.

**Uses of hashing:**
- **Storing passwords:** systems store the **hash** of a password, not the password itself. When a user logs in, the entered password is hashed and compared to the stored hash. If the database is stolen, attackers do not get the actual passwords.
- **Hash tables:** a data structure that maps a **key** to an **index** in a table using a hash function, giving very fast (close to O(1)) lookup, insertion and deletion.

> 💡 **A\* extension:** A **collision** occurs when two keys hash to the same index/value. Hash tables must handle collisions, e.g. by **chaining** (storing a list at each index) or **probing** (looking for the next free slot). Encryption and hashing are not the same: encryption is reversible (with a key); hashing is one-way and not designed to be reversed.

## 1.3.2 Databases

### Flat file vs relational databases

**Database** — an organised, persistent store of data.

**Flat file database** — a database held in a single table/file. Simple, but leads to **data redundancy** (the same data repeated many times) and **data inconsistency / update anomalies** (having to change the same fact in many places).

**Relational database** — data is split across multiple linked **tables**, removing redundancy. Tables are linked using keys. Easier to maintain, more scalable, reduces redundancy and supports complex queries.

| Flat file | Relational |
|---|---|
| Single table | Multiple linked tables |
| High data redundancy | Reduced redundancy |
| Prone to inconsistency/anomalies | Consistency through normalisation |
| Simple to set up | More complex but scalable |

### Key terms: entities, attributes, records

- **Entity** — a category of thing about which data is stored (e.g. *Student*, *Book*). Usually implemented as a table.
- **Attribute / field** — a property of an entity (a column, e.g. *Surname*).
- **Record / row / tuple** — all the data about one instance of an entity.

### Keys

- **Primary key** — an attribute (or set of attributes) that **uniquely identifies each record** in a table.
- **Foreign key** — an attribute in one table that is the **primary key in another table**; it creates the link between the two tables.
- **Composite key** — a primary key made up of **two or more attributes** combined, because no single attribute is unique on its own.
- **Secondary key** — an attribute used to allow records to be **searched/indexed** quickly even though it isn't the primary key.

### Entity relationship (ER) modelling

An **ER diagram** shows entities and the **relationships** between them. Relationships can be:
- **One-to-one (1:1)**
- **One-to-many (1:M)**
- **Many-to-many (M:N)**

A many-to-many relationship cannot be implemented directly in a relational database — it is resolved by introducing a **linking (junction) table**, typically with a composite key.

OCR notation: entities in boxes, lines between them, with "crow's foot" or 1 / M labels showing the degree of the relationship.

### Normalisation

**Normalisation** is the process of organising data in a database to **reduce redundancy** and remove **anomalies**, while keeping data consistent.

**First normal form (1NF):** no **repeating groups** of attributes; each field contains a single (atomic) value; there is a primary key.

**Second normal form (2NF):** is in 1NF **and** every non-key attribute depends on the **whole** primary key (remove **partial dependencies** — only relevant when there is a composite key).

**Third normal form (3NF):** is in 2NF **and** there are no **non-key dependencies** (no non-key attribute depends on another non-key attribute — remove **transitive dependencies**).

> 💡 **A\* extension:** A handy summary is that, in 3NF, every non-key attribute depends on *"the key, the whole key, and nothing but the key."* "The key" = 1NF (atomic, has a key), "the whole key" = 2NF (no partial dependency), "nothing but the key" = 3NF (no transitive dependency).

### Referential integrity

**Referential integrity** ensures that relationships between tables remain consistent: a **foreign key value must always refer to a valid existing primary key** in the related table (or be null). This prevents "orphaned" records — e.g. you cannot record an order for a customer who does not exist, and you cannot delete a customer who still has orders.

### SQL

**SQL (Structured Query Language)** is used to define, query and manipulate relational databases.

```sql
-- Retrieve data
SELECT Surname, Forename
FROM Student
WHERE YearGroup = 13
ORDER BY Surname ASC;

-- Join two tables
SELECT Student.Surname, Class.ClassName
FROM Student
JOIN Class ON Student.ClassID = Class.ClassID;

-- Insert a new record
INSERT INTO Student (StudentID, Surname, Forename, YearGroup)
VALUES (1024, 'Patel', 'Aisha', 12);

-- Update existing records
UPDATE Student
SET YearGroup = 13
WHERE YearGroup = 12;

-- Delete records
DELETE FROM Student
WHERE StudentID = 1024;

-- Create a new table
CREATE TABLE Student (
  StudentID  INTEGER PRIMARY KEY,
  Surname    VARCHAR(30),
  Forename   VARCHAR(30),
  YearGroup  INTEGER
);
```

Useful clauses/keywords: `SELECT`, `FROM`, `WHERE`, `JOIN ... ON`, `ORDER BY` (`ASC`/`DESC`), `INSERT INTO ... VALUES`, `UPDATE ... SET`, `DELETE FROM`, `CREATE TABLE`, `AND`/`OR`, `LIKE` (with wildcards `%`/`*`), `*` (all fields).

> 💡 **A\* extension:** `WHERE` filters which rows are returned; `JOIN` combines columns from related tables on a matching key. `SELECT *` returns all columns — fine for revision, but in real systems you select only the columns you need to reduce data transfer. `LIKE 'A%'` finds values starting with "A".

### Transaction processing and ACID

A **transaction** is a single logical operation made up of one or more steps that must **all succeed or all fail** (e.g. transferring money: debit one account *and* credit another).

**ACID** describes the properties a reliable transaction must have:

| Property | Meaning |
|---|---|
| **Atomicity** | The transaction is all-or-nothing: if any part fails, the whole transaction is rolled back so no partial change is made. |
| **Consistency** | A transaction takes the database from one valid state to another, never breaking rules/constraints (e.g. referential integrity). |
| **Isolation** | Concurrent transactions do not interfere with each other; the result is as if they ran one after another. |
| **Durability** | Once a transaction is committed, it is permanently saved, even if there is a power/system failure. |

**Record locking** — while one user/transaction is updating a record, it is **locked** so other users cannot edit it at the same time. This prevents the **lost update problem** (where two concurrent edits overwrite each other). A risk of locking is **deadlock**, where two transactions each wait for a record the other has locked.

**Redundancy** (in this context) means keeping **duplicate copies of data/hardware** (e.g. backup servers, RAID, mirrored databases) so that if one fails the system continues — improving reliability and availability. (Contrast with *data redundancy*, the wasteful repetition that normalisation removes.)

> 💡 **A\* extension:** Distinguish the two meanings of "redundancy" carefully. Normalisation removes **unnecessary data redundancy** (repeated data), but deliberate **hardware/data redundancy** (backups, failover servers) is desirable for fault tolerance and supports the **Durability** property of ACID.

## 1.3.3 Networks

### Characteristics of networks

A **network** is two or more devices connected so they can communicate and share resources (files, printers, internet access). Benefits: sharing resources/data, central management and backup, communication. Drawbacks: cost, security risks, dependence on the network, malware can spread.

**LAN (Local Area Network)** — covers a **small geographical area** (one site/building), usually owned and maintained by one organisation, often with its own cabling/hardware. High speed.

**WAN (Wide Area Network)** — covers a **large geographical area** (between cities/countries), uses **third-party** communication infrastructure (e.g. leased lines, fibre). The internet is the largest WAN.

### Client-server vs peer-to-peer

**Client-server** — one or more powerful **servers** provide resources/services to **client** devices that request them. Central control, easier backup and security, but server is a single point of failure and can be expensive.

**Peer-to-peer (P2P)** — devices have **equal status**; each can act as both client and server, sharing resources directly. Cheaper, no dependence on a central server, but harder to manage, back up and secure.

| Client-server | Peer-to-peer |
|---|---|
| Central server provides services | All peers equal |
| Centralised security/backup | Decentralised |
| Server = single point of failure | No single point of failure |
| Scales well, more costly | Cheaper, harder to manage |

### Structure of the internet

The **internet** is a global network of interconnected networks ("network of networks"). Key parts:
- **Backbone** — high-capacity data routes connecting major networks worldwide.
- **ISP (Internet Service Provider)** — a company that provides users/organisations with access to the internet.
- **IP addressing** — every device on a network has an **IP address** that uniquely identifies it so data can be routed to it (IPv4 = 32-bit, e.g. `192.168.0.1`; IPv6 = 128-bit).
- **DNS (Domain Name System)** — a system of servers that translates human-readable **domain names** (e.g. `www.ocr.org.uk`) into the **IP addresses** computers use to locate each other.

> 💡 **A\* extension:** The DNS is hierarchical. A query goes to a resolver, then potentially to **root**, **top-level domain (TLD, e.g. .uk)** and **authoritative** name servers until the IP is found. Results are cached to speed up future lookups. Without DNS, users would have to remember numeric IP addresses.

### Network topologies

**Topology** = the layout/arrangement of devices on a network.

**Star topology** — every device connects to a **central node** (switch/hub).
- Advantages: if one cable fails only that device is affected; good performance; easy to add devices.
- Disadvantage: if the central node fails the whole network fails; needs more cabling.

**Mesh topology** — devices are interconnected, with **multiple paths** between nodes (full mesh = every node connected to every other).
- Advantages: very resilient/fault tolerant (alternative routes); no single point of failure; good for wireless.
- Disadvantage: expensive (lots of connections in a wired full mesh).

> 💡 **A\* extension:** A **physical** topology (how cables are actually laid) can differ from the **logical** topology (how data appears to flow). A network physically wired as a star can behave logically as a bus, for example.

### Packet switching vs circuit switching

**Packet switching** — data is split into **packets**, each with a header (source/destination IP, sequence number, etc.). Packets are sent **independently** and may take **different routes** through the network, then are reassembled in order at the destination.
- Efficient use of the network; resilient (can route around failures); no dedicated connection needed.

**Circuit switching** — a **dedicated physical communication path** (circuit) is set up between sender and receiver for the whole duration of the communication (e.g. traditional telephone calls).
- Guarantees bandwidth and constant connection, but the circuit is reserved even when idle, so it is **less efficient**.

| Packet switching | Circuit switching |
|---|---|
| Data split into packets | Dedicated circuit |
| Packets routed independently | One fixed path |
| Efficient, resilient | Reserved bandwidth, can waste capacity |
| Used by the internet | Traditional telephone networks |

### Protocols and standards

**Protocol** — a set of rules governing how devices communicate (format, timing, error handling).
**Standard** — an agreed specification that ensures different manufacturers' hardware/software are **compatible and interoperable**.

### The TCP/IP stack

The **TCP/IP stack** is a layered model. Each layer has a specific job and adds/removes its own header. Layering means each layer can be changed independently as long as the interface stays the same.

| Layer | Role | Example protocols |
|---|---|---|
| **Application** | Provides services to user applications and prepares data | HTTP, HTTPS, FTP, SMTP, POP3, IMAP |
| **Transport** | Splits data into packets, manages end-to-end delivery, error checking, reassembly; adds port numbers | TCP, UDP |
| **Network / Internet** | Adds source and destination **IP addresses**; routes packets across networks | IP |
| **Link / Data link** | Handles the physical connection; adds **MAC addresses** for the next hop | Ethernet, Wi-Fi |

Sending: data passes **down** the stack (Application → Link), each layer adding a header. Receiving: data passes **up** the stack, each layer removing its header.

### Common protocols

| Protocol | Purpose |
|---|---|
| **HTTP** | HyperText Transfer Protocol — transfers web pages between web server and browser. |
| **HTTPS** | Secure HTTP — HTTP encrypted (TLS/SSL) for secure web communication. |
| **FTP** | File Transfer Protocol — transfers/uploads/downloads files between computers. |
| **SMTP** | Simple Mail Transfer Protocol — **sending** email (and server-to-server transfer). |
| **POP3** | Post Office Protocol — **retrieving** email; typically downloads and removes mail from the server. |
| **IMAP** | Internet Message Access Protocol — **retrieving** email; mail stays on the server and is synced across devices. |

### Network hardware

- **NIC (Network Interface Card)** — hardware that connects a device to a network; holds the device's unique **MAC address**.
- **MAC address** — a unique 48-bit physical hardware address assigned to a NIC; used at the link layer to identify devices on a local network. (Unlike an IP address it does not normally change.)
- **Switch** — connects devices on a LAN and sends data only to the specific device it is intended for (using MAC addresses).
- **Router** — connects different networks together and routes packets between them using IP addresses (e.g. connecting a LAN to the internet).
- **WAP (Wireless Access Point)** — allows wireless devices to connect to a wired network.

### Network security: firewalls and proxies

**Firewall** — hardware or software that monitors and controls traffic entering/leaving a network based on rules, using **packet filtering** and often a **stateful inspection**, to block unauthorised access.

**Proxy server** — acts as an **intermediary** between clients and the internet. It can hide users' IP addresses (privacy/anonymity), **cache** frequently requested pages (speed/reduced bandwidth), and filter/log requests (content control).

> 💡 **A\* extension:** A firewall typically uses two NICs and acts as a checkpoint; **packet filtering** inspects packet headers (source/destination IP, port) against an access control list. Encryption protects data **in transit** so even if intercepted it cannot be read. Firewalls, proxies and encryption tackle different threats and are usually combined (defence in depth).

## 1.3.4 Web technologies

### HTML, CSS and JavaScript

**HTML (HyperText Markup Language)** — defines the **structure and content** of a web page (headings, paragraphs, links, images).

```html
<!DOCTYPE html>
<html>
  <head><title>My Page</title></head>
  <body>
    <h1 id="main">Hello</h1>
    <p class="intro">Welcome to my site.</p>
    <button onclick="greet()">Click me</button>
  </body>
</html>
```

**CSS (Cascading Style Sheets)** — defines the **presentation/style** of a page (colours, fonts, layout). Separating style from content makes sites easier to maintain and consistent. Selectors target tags, **classes** (`.intro`) and **ids** (`#main`).

```css
h1 { color: navy; font-family: Arial; }
.intro { font-size: 16px; }
#main { text-align: center; }
```

**JavaScript** — a scripting language that adds **interactivity and behaviour**, usually run in the browser (**client-side**) to respond to events, validate forms, update the page dynamically.

```javascript
function greet() {
  alert("Hello, visitor!");
}
```

> 💡 **A\* extension:** Selector specificity matters: an `id` (`#main`) is more specific than a `class` (`.intro`), which is more specific than an element selector (`h1`). External CSS files let one stylesheet control an entire site, so a single change updates every page.

### Search engine indexing

A **web crawler / spider** automatically follows links across the web and reads pages. The search engine builds an **index**: a data structure mapping **keywords** to the pages that contain them (like a book's index), so that when a user searches, results can be returned quickly without scanning the whole web. Pages are then **ranked** for relevance.

### The PageRank algorithm

**PageRank** is an algorithm (originally used by Google) that ranks web pages by **importance**, mainly based on the **links pointing to them**.

Conceptually:
- Each link to a page is treated like a **"vote"** of confidence.
- A link from an **important** page (one that itself has a high PageRank) is worth **more** than a link from an unimportant page.
- A page that links to **many** pages spreads its "vote" thinly — each outgoing link passes on a fraction of its PageRank (its rank divided by the number of outgoing links).
- A **damping factor** (often ~0.85) models the chance a "random surfer" keeps clicking links rather than jumping to a random page, and stops ranks growing without bound.
- The calculation is **iterative**: PageRank values are recalculated repeatedly until they **converge** (stop changing significantly).

So a page is ranked highly if many pages — especially important ones — link to it.

> 💡 **A\* extension:** PageRank uses links, not just keyword matching, so it is harder to manipulate by stuffing a page with keywords. The random surfer model and damping factor guarantee the iterative process converges. PageRank is only one signal; real search ranking also uses content relevance, freshness and many other factors.

### Server-side vs client-side processing

**Client-side processing** — code runs in the **user's browser** (e.g. JavaScript).
- Advantages: fast response (no round trip to server), reduces server load, immediate feedback (e.g. form validation).
- Disadvantages: depends on the user's device/browser; can be **viewed and disabled/bypassed** by the user, so it is **not secure** on its own.

**Server-side processing** — code runs on the **web server** (e.g. PHP, querying a database).
- Advantages: more **secure** (code/data hidden from user), can access server databases, not dependent on client device.
- Disadvantages: increases server load; slower due to network round trips.

| | Client-side | Server-side |
|---|---|---|
| Where it runs | User's browser | Web server |
| Speed/responsiveness | Fast, immediate | Slower (round trip) |
| Security | Insecure, user can see/bypass | Secure, hidden |
| Server load | Reduced | Increased |
| Example | JavaScript form validation | Database query, login check |

> 💡 **A\* extension:** A common exam scenario is form validation. **Client-side** validation gives instant feedback and reduces wasted server requests, but a malicious user can bypass it. Therefore important checks (e.g. login, payment) **must also** be validated server-side. Best practice is to do both.

## Worked examples

### Worked example 1 — Run length encoding (RLE)

Consider a row of pixels where each character is a colour:

```
WWWWWWWWWWWWBWWWWWWWWWWWWBBB
```

Count each run (value + number of repeats):

```
12W 1B 12W 3B
```

Encoded as pairs (count, value): `12W1B12W3B`.

- Original: 27 characters.
- Encoded: stores 4 runs (8 symbols).
- This is **lossless** — the exact original can be reconstructed.

**Why it can fail:** for data with no repeats, e.g. `ABCDEF`, RLE gives `1A1B1C1D1E1F`, which is **larger** than the original. RLE only saves space when there are long runs.

### Worked example 2 — Normalisation (UNF → 1NF → 2NF → 3NF)

Start with an **unnormalised** flat table storing student orders for revision guides:

| OrderID | StudentName | TutorGroup | Books (Title, Price)... |
|---|---|---|---|

Problems: repeating group of books, redundant data (tutor group repeated), mixed entities.

**1NF — remove repeating groups; atomic values; primary key.** Each row holds one book ordered:

`Order(OrderID, StudentID, StudentName, TutorGroup, BookID, BookTitle, BookPrice, Quantity)`
Primary key: composite `(OrderID, BookID)`.

**2NF — remove partial dependencies (depend on the whole composite key).**
- `BookTitle`, `BookPrice` depend only on `BookID` (part of the key) → move to a `Book` table.
- `StudentName`, `TutorGroup` depend only on `OrderID` (via the student) → move out.

```
Order(OrderID, StudentID, ...)
OrderLine(OrderID, BookID, Quantity)      -- composite key
Book(BookID, BookTitle, BookPrice)
```

**3NF — remove transitive dependencies (non-key → non-key).**
Suppose each `TutorGroup` determines a `Teacher`. `Teacher` depends on `TutorGroup`, not directly on the student key → split it out:

```
Student(StudentID, StudentName, TutorGroup*)
TutorGroup(TutorGroup, Teacher)
Order(OrderID, StudentID*, OrderDate)
OrderLine(OrderID*, BookID*, Quantity)
Book(BookID, BookTitle, BookPrice)
```

(`*` = foreign key.) Now every non-key attribute depends on **the key, the whole key, and nothing but the key.** Redundancy and update anomalies are removed.

### Worked example 3 — SQL query

"List the surname and tutor group of all Year 13 students whose surname begins with 'M', sorted alphabetically."

```sql
SELECT Surname, TutorGroup
FROM Student
WHERE YearGroup = 13 AND Surname LIKE 'M%'
ORDER BY Surname ASC;
```

### Worked example 4 — PageRank explained simply

Imagine pages A, B, C, D.
- B, C and D **all link to A** → A receives three "votes", so A is likely important.
- C is itself linked to by many pages, so C has a **high** PageRank; its vote for A counts for a lot.
- D links to 10 different pages, so D's vote is **divided 10 ways** — each page D links to gets only a small share.
- The algorithm repeats this calculation **iteratively** until the values **converge**.

Result: A ranks highly because it is linked to by **many** pages, including at least one **important** page (C).

## Common exam mistakes & examiner tips

- **"Lossy can be reversed"** — wrong. Lossy permanently removes data; you cannot recover the exact original. Don't confuse "reduced quality" with "recoverable".
- **RLE always shrinks files** — wrong. It can *enlarge* files with little repetition. State *when* it is effective (long runs).
- **Symmetric vs asymmetric** — be precise: symmetric = *same* key (key distribution problem); asymmetric = *public + private* pair. Don't say "asymmetric is just two keys" without explaining the public/private roles.
- **Hashing = encryption** — wrong. Hashing is **one-way**; encryption is reversible with a key. Passwords are **hashed**, not encrypted, for storage.
- **Primary vs foreign key** — the foreign key is the *primary key of another table*. Always say which table.
- **Normalisation** — explain *why* (remove redundancy/anomalies), not just "splits into tables". Name the dependency removed at each stage (repeating group → 1NF, partial → 2NF, transitive → 3NF).
- **ACID** — learn all four words and what each prevents; don't just list the letters.
- **Switch vs router** — switch connects devices *within* a LAN (uses MAC); router connects *different networks* (uses IP). Don't swap them.
- **Two meanings of "redundancy"** — data redundancy (bad, removed by normalisation) vs backup/hardware redundancy (good, for reliability). Read the question context.
- **Client-side validation is "secure"** — wrong. It can be bypassed; security-critical checks must be server-side.
- **DNS "stores websites"** — no, DNS translates domain names to IP addresses.
- For SQL questions, watch the **exact** clause order: `SELECT ... FROM ... WHERE ... ORDER BY`. Include the table name in joins.

## Exam-style questions

**Q1 (AO1, 2 marks).** State two differences between lossy and lossless compression.

<details><summary>Mark scheme</summary>

1 mark each (max 2):
- Lossy permanently removes data / lossless does not.
- With lossless the original can be recovered exactly / with lossy it cannot.
- Lossy gives (generally) greater size reduction than lossless.
- Lossy used for images/audio/video; lossless for text/program files.
</details>

**Q2 (AO1, 4 marks).** Explain the difference between symmetric and asymmetric encryption, and give one advantage of asymmetric encryption.

<details><summary>Mark scheme</summary>

- Symmetric uses the same (single) key to encrypt and decrypt (1).
- Asymmetric uses a pair of keys — a public key and a private key (1).
- Data encrypted with the public key can only be decrypted with the matching private key (1).
- Advantage: solves the key distribution problem / the public key can be shared openly without compromising security (1).
</details>

**Q3 (AO2, 3 marks).** A simple black-and-white image contains a row of pixels: `BBBBBWWWBB`. Show how run length encoding would encode this row and explain why RLE is suitable here.

<details><summary>Mark scheme</summary>

- Correct encoding, e.g. `5B 3W 2B` / `5B3W2B` (1).
- Identifies it as lossless / original fully recoverable (1).
- Suitable because there are runs of repeated identical pixels, so it saves space (1).
</details>

**Q4 (AO1/AO2, 6 marks).** A school stores all data in a single flat-file table and suffers from data redundancy. Describe how normalising the data to third normal form (3NF) would address this. Refer to 1NF, 2NF and 3NF.

<details><summary>Mark scheme</summary>

Up to 6 marks; reward stages and reasoning:
- Flat file has redundancy / update and deletion anomalies; normalisation splits data into linked tables to remove this (1).
- 1NF: remove repeating groups; ensure atomic values; define a primary key (1).
- 2NF: in 1NF and remove partial dependencies — every non-key attribute depends on the whole (composite) key (1).
- 3NF: in 2NF and remove transitive/non-key dependencies — no non-key attribute depends on another non-key attribute (1).
- Tables are linked using primary and foreign keys / referential integrity maintained (1).
- Result: reduced redundancy, no update/insert/delete anomalies, consistent data (1).
</details>

**Q5 (AO2, 4 marks).** Write an SQL statement to display the `Forename` and `Email` of all members in the `Member` table who live in `Town = 'Oxford'`, sorted by `Forename` in ascending order.

<details><summary>Mark scheme</summary>

```sql
SELECT Forename, Email
FROM Member
WHERE Town = 'Oxford'
ORDER BY Forename ASC;
```
- Correct `SELECT` of the two named fields (1).
- Correct `FROM Member` (1).
- Correct `WHERE Town = 'Oxford'` (1).
- Correct `ORDER BY Forename` (ASC) (1).
</details>

**Q6 (AO1, 4 marks).** Describe the role of the four layers of the TCP/IP stack when sending data.

<details><summary>Mark scheme</summary>

1 mark per layer correctly described (max 4):
- Application: provides services to applications / prepares data using protocols such as HTTP, SMTP (1).
- Transport: splits data into packets, manages end-to-end delivery and reassembly, adds port numbers (TCP) (1).
- Network/Internet: adds source and destination IP addresses and routes packets (1).
- Link: handles physical transmission and adds MAC addresses (1).
</details>

**Q7 (AO2, 4 marks).** Compare circuit switching and packet switching.

<details><summary>Mark scheme</summary>

Up to 4 marks (comparison):
- Circuit switching reserves a dedicated path for the whole communication; packet switching does not (1).
- Packets are routed independently and can take different routes / reassembled at destination (1).
- Packet switching uses the network more efficiently / no bandwidth wasted when idle (1).
- Packet switching is more resilient — can route around failures; circuit switching guarantees bandwidth/constant connection (1).
</details>

**Q8 (AO3, 9 marks).** An online shop processes payments through its website. Evaluate the use of client-side and server-side processing for handling and validating customer payment data. (You may also refer to encryption and hashing.)

<details><summary>Mark scheme</summary>

Levelled response (indicative content; banded 0–9):

**AO1 (knowledge):**
- Client-side runs in the browser (e.g. JavaScript); server-side runs on the web server.
- Encryption (HTTPS) protects data in transit; hashing stores passwords one-way.

**AO2 (application):**
- Client-side validation gives instant feedback and reduces unnecessary server requests/load.
- Server-side processing can securely access the database and is hidden from the user.

**AO3 (evaluation/judgement):**
- Client-side validation can be viewed/bypassed/disabled, so it is not secure on its own.
- Therefore payment data must also be validated and processed server-side.
- HTTPS (asymmetric to exchange a key, then symmetric) should encrypt data in transit; passwords stored as hashes.
- Justified conclusion: use client-side for responsiveness but rely on server-side + encryption for security; best practice is to combine both.

**Level 3 (7–9):** thorough, balanced, well-applied to the payment scenario with a justified conclusion.
**Level 2 (4–6):** some application and comparison, conclusion may be limited.
**Level 1 (1–3):** basic descriptive points, little application/evaluation.
</details>
