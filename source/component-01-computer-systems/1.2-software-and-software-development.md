## 1.2.1 Operating systems

The **operating system (OS)** is the suite of software that manages the hardware and provides a platform on which application software can run. It sits between the user/applications and the hardware, hiding complexity (this is called **abstraction**).

Core functions of an OS:
- **Memory management** — allocating memory to programs and keeping them separate.
- **Processor (CPU) scheduling** — deciding which process gets the CPU and when.
- **Interrupt handling** — responding to hardware/software signals.
- **Managing I/O / peripherals** — via device drivers.
- **Providing a user interface** (GUI or CLI).
- **Security and file management** — user accounts, permissions, the file system.

### Memory management

A computer runs many programs at once but has limited RAM. The OS must share RAM fairly, keep processes from corrupting each other's memory, and allow programs larger than physical RAM to run.

**Paging** is dividing memory into **fixed-size** blocks. Physical memory is split into **frames** and the logical program is split into equal-sized **pages**. Pages are loaded into any free frame, so a program need not occupy a contiguous block. A **page table** maps logical pages to physical frames.

**Segmentation** is dividing memory into **variable-size** blocks called **segments** that reflect the logical structure of the program (e.g. a segment for the main code, one for a subroutine, one for data). Segments vary in size because they match logical divisions rather than fixed lengths.

| | Paging | Segmentation |
|---|---|---|
| Block size | Fixed | Variable |
| Divided by | OS (physical) | Logical structure of program |
| Suffers from | Internal fragmentation | External fragmentation |
| Visible to programmer? | No (transparent) | Often reflects program structure |

**Virtual memory** uses a section of secondary storage (the **swap space** / page file) as if it were extra RAM. Pages not currently needed are written out to disk; when they are needed again they are paged back in. This lets programs run that are larger than physical RAM, and lets more programs be open at once.

- When a program needs a page that is on disk, a **page fault** occurs and the OS swaps it back into RAM.
- If too little RAM is available, the OS spends most of its time swapping pages in and out instead of doing useful work — this is **disk thrashing**, and it makes the machine very slow.

> 💡 **A\* extension**
> Virtual memory works because of the **principle of locality of reference** — a program tends to access a small set of pages repeatedly over a short time, so only those need to be in RAM. The OS chooses which page to evict using a replacement policy (e.g. *least recently used*). On modern CPUs a **Translation Lookaside Buffer (TLB)** caches recent page-table lookups so address translation does not cost an extra memory access every time. Be ready to argue the trade-off: virtual memory increases the *amount* of programs you can run but the slow disk access means it is far slower than real RAM.

### Interrupts and the interrupt service routine

An **interrupt** is a signal generated by hardware or software that tells the processor that a device or process needs attention. Interrupts let the CPU respond to events without wasting time **polling** (constantly checking) every device.

Examples: a key pressed, a disk finishing a transfer, a hardware fault, a divide-by-zero error, a timer signal (used by the scheduler), a print job needing data.

Interrupts have different **priorities**, often held in a **priority queue / interrupt register**. A high-priority interrupt can interrupt the handling of a lower-priority one.

How the processor handles an interrupt:
1. At the **end of each fetch–decode–execute cycle**, the CPU checks the interrupt register/lines.
2. If an interrupt is present (and is higher priority than the current task), the CPU finishes the current instruction.
3. The contents of registers (PC, accumulator, etc.) are **pushed onto the stack** so the current job can resume later.
4. The processor loads the address of the appropriate **interrupt service routine (ISR)** — a small program that deals with that specific interrupt.
5. The ISR runs to service the device/event.
6. When finished, the saved register contents are **popped off the stack** and the CPU resumes the interrupted task exactly where it left off.

The **interrupt service routine (ISR)** (also called the interrupt handler) is the dedicated piece of code that the OS runs to deal with a particular type of interrupt.

> 💡 **A\* extension**
> Lower-priority interrupts are not lost; they are held until the higher-priority ISR completes. Because register contents are saved on a **stack** (LIFO), interrupts can themselves be interrupted and still resume correctly in the right order — this is *nested interrupt handling*.

### Scheduling

**Scheduling** is the OS deciding which process gets CPU time and for how long, to give the illusion that many processes run simultaneously on a single core, and to use the CPU efficiently. Good scheduling aims to: maximise CPU use, be fair, give good response times, and meet deadlines.

Scheduling algorithms are either **pre-emptive** (a running process can be stopped and CPU reallocated) or **non-pre-emptive** (a process runs until it finishes or blocks).

| Algorithm | How it works | Pre-emptive? | Notes |
|---|---|---|---|
| **Round robin** | Each process gets a fixed **time slice** (quantum) in turn; if not finished it goes to the back of the queue | Yes | Fair; no starvation; ignores priority/length |
| **First come first served (FCFS)** | Processes run in arrival order until complete | No | Simple; long jobs delay short ones (**convoy effect**) |
| **Shortest job first (SJF)** | The waiting process with the shortest **total** runtime runs next | No | Minimises average wait; needs runtime known in advance; long jobs may **starve** |
| **Shortest remaining time (SRT)** | The process with the shortest **remaining** time runs next; a newly arrived shorter job pre-empts | Yes | Good response for short jobs; long jobs may starve; overhead of switching |
| **Multi-level feedback queue (MLFQ)** | Several queues of different priority; jobs move between queues based on behaviour (e.g. a CPU-hungry job is demoted to a lower-priority queue) | Yes | Flexible, balances responsiveness and throughput; complex to tune |

> 💡 **A\* extension**
> SJF and SRT need the burst time to be *estimated* — the OS cannot truly know how long a job will take, so it predicts from past behaviour. MLFQ effectively *approximates* SJF without needing to know runtimes: short, interactive jobs naturally stay in high-priority queues while CPU-bound jobs sink to lower ones. In an exam, link the choice of algorithm to the OS *type* — a real-time OS needs predictable, deadline-driven scheduling, not plain round robin.

### Types of operating system

| Type | Description | Example use |
|---|---|---|
| **Distributed OS** | Runs across **multiple networked computers**, sharing the workload and presenting them as one system | Server clusters, large-scale computation |
| **Embedded OS** | Built for a **specific device** with limited resources; held in ROM, efficient, reliable | Washing machine, smart TV, car engine controller |
| **Multitasking OS** | Allows several programs (processes) to run apparently at once by rapidly switching the CPU between them (time-slicing) | Windows, macOS, Linux desktops |
| **Multi-user OS** | Allows **several users** to use the machine's resources at the same time; the scheduler shares CPU time and the OS manages permissions | Mainframes, university/cloud servers |
| **Real-time OS (RTOS)** | **Guarantees a response within a set time limit**; processing must happen by a deadline | Aircraft control, pacemaker, anti-lock brakes |

### BIOS

The **BIOS (Basic Input Output System)** is firmware stored on a ROM/flash chip on the motherboard. When the computer is switched on, the BIOS is the **first program to run**. It:
- Runs the **POST (Power-On Self-Test)** to check hardware (RAM, keyboard, storage) is present and working.
- Initialises hardware and stores low-level settings (system clock, boot order).
- Locates the boot device and **loads the operating system** (the bootstrap/boot loader) into RAM, then hands over control.

### Device drivers

A **device driver** is software that allows the operating system to communicate with and control a specific piece of hardware (e.g. printer, graphics card, mouse). The driver translates generic OS commands into the specific instructions that the device understands. Drivers are **hardware- and OS-specific**, which is why a new driver is needed for each device, and updates are released for new OS versions.

### Virtual machines and intermediate code

A **virtual machine (VM)** is software that emulates (acts like) a complete computer system. There are two exam contexts:

1. **System virtual machine** — software that lets one physical computer run one or more *complete* operating systems as guests (e.g. running Linux inside Windows). Useful for testing, security/sandboxing, and running legacy software.

2. **Process virtual machine / intermediate code** — source code is compiled not to native machine code but to platform-independent **intermediate code** (e.g. Java **bytecode**). A VM (e.g. the Java Virtual Machine) then runs that intermediate code on any device that has the VM installed.

**Intermediate code** is code halfway between source code and machine code; it is independent of any particular processor.

- **Advantage:** portability — "write once, run anywhere"; the same intermediate code runs on any platform with the right VM.
- **Disadvantage:** it runs slower than native machine code because the VM must translate/interpret the intermediate code at run time, and the VM itself uses resources.

## 1.2.2 Applications generation

### Nature of applications and utilities

**Application software** is software designed to let the user complete a specific real-world task (e.g. word processor, web browser, spreadsheet, game). The user runs it directly to get a job done.

**Utility software** is system software that helps maintain, configure or optimise the computer itself (rather than doing the user's work). Examples: disk defragmenter, backup tool, antivirus, file compression, disk clean-up, firewall.

### Open source vs closed source

| | Open source | Closed source (proprietary) |
|---|---|---|
| **Source code** | Available, can be viewed and modified | Hidden; cannot legally be modified |
| **Cost** | Usually free | Usually paid (licence) |
| **Support** | Community-based | Official paid support from the vendor |
| **Modification/redistribution** | Permitted under the licence | Forbidden by licence |
| **Examples** | Linux, Firefox, GIMP | Microsoft Windows, Adobe Photoshop |

> 💡 **A\* extension**
> Open source is not the same as "freeware" — freeware is free of charge but its source code stays hidden. Open source's strength is that many developers can inspect and improve it (often improving security), but a business may prefer closed source for **guaranteed support**, accountability and a polished, supported product. Judge each by the *context* in the question (budget, in-house expertise, need for support).

### Translators

A **translator** converts program code from one form to another. Three types:

**Assembler** — translates **assembly language** into **machine code**. Generally there is a one-to-one relationship between an assembly instruction (mnemonic) and a machine code instruction.

**Compiler** — translates **high-level source code into machine code all at once**, producing a standalone **executable** file before the program is run.

**Interpreter** — translates and executes **high-level source code one line/statement at a time** at run time; it does not produce a separate executable.

| | Compiler | Interpreter |
|---|---|---|
| When translated | All at once, before running | Line by line, at run time |
| Output | Standalone executable file | No standalone file produced |
| Speed of final program | Fast (already machine code) | Slower (translated each run) |
| Source code needed to run? | No (distribute the .exe only) | Yes, plus an interpreter |
| Error reporting | Reports all errors at end of compilation | Stops at the first error it reaches |
| Portability | Executable is platform-specific | Portable if interpreter exists for platform |
| Good for | Distributing finished software | Developing/debugging, scripting |

> 💡 **A\* extension**
> Compiling protects intellectual property because only machine code is shared, not source. Interpreters suit development because you get immediate feedback line by line. Many modern languages combine both: Python *compiles* to bytecode then *interprets* it, and Java compiles to bytecode that is **JIT (just-in-time) compiled** by the VM at run time — link this straight back to intermediate code in 1.2.1.

### Stages of compilation

A compiler works in four stages:

1. **Lexical analysis** — the source code is read and broken into **tokens** (keywords, identifiers, operators). Whitespace and comments are removed. A **symbol table** is created to hold identifiers (variables, subroutines) and their attributes.

2. **Syntax analysis (parsing)** — the tokens are checked against the **grammar/rules** of the language. A parse/syntax tree is built. **Syntax errors** are reported here (e.g. a missing bracket). Entries are added to the symbol table.

3. **Code generation** — the parse tree and symbol table are used to produce the **object code (machine code)** equivalent of the program.

4. **(Code) optimisation** — the generated code is refined to **run faster and/or use less memory** (e.g. removing redundant instructions, simplifying loops) while producing the same result. (Note: optimisation can also be considered to take place around/with code generation.)

> 💡 **A\* extension**
> Optimisation is a trade-off: aggressive optimisation makes the program faster/smaller but the compilation itself takes longer and the code becomes harder to debug. Lexical analysis only checks that tokens are *valid*; it cannot spot grammar errors — that is the job of syntax analysis. Semantic checks (e.g. type mismatches) happen alongside/after syntax analysis.

### Linkers, loaders and libraries

A **library** is a pre-written, pre-compiled and tested collection of subroutines/modules that programmers can reuse (e.g. a maths library, a graphics library). Using libraries saves development time and increases reliability.

A **linker** combines the compiled program with the library modules (and other separately compiled modules) it uses into a single executable file, resolving the references between them.

- **Static linking** copies the library code *into* the executable — larger file, but self-contained.
- **Dynamic linking** keeps the library separate (e.g. a `.dll`) and links it at run time — smaller executable, libraries shared and updated independently.

A **loader** is part of the OS; it copies the executable program from secondary storage **into RAM** and prepares it to run (e.g. setting addresses), then execution begins.

## 1.2.3 Software development

The **software development lifecycle** describes the stages a project passes through. A **methodology** is the overall approach taken to move through those stages.

### Methodologies

**Waterfall** — a **linear, sequential** model: each stage (analysis → design → implementation → testing → evaluation/maintenance) is completed before the next begins, flowing downwards like a waterfall.
- *Pros:* simple, well-documented, clear milestones; easy to manage.
- *Cons:* inflexible — hard to go back and change requirements; the client only sees the product at the end, so misunderstandings surface late.
- *Use when:* requirements are well understood and fixed, and the project is small/straightforward.

**Agile** — an umbrella of **iterative** approaches that deliver working software in small, frequent increments with continuous client feedback.
- *Pros:* flexible to changing requirements; client involved throughout; working software delivered early and often.
- *Cons:* less documentation; harder to predict final cost/time; needs close client availability.
- *Use when:* requirements are likely to change or are unclear at the start.

**Extreme programming (XP)** — an agile method emphasising code quality and teamwork: **pair programming**, frequent small releases, continuous testing (test-driven development), and the customer is part of the team.
- *Pros:* very high code quality; responsive to change; constant testing reduces bugs.
- *Cons:* demanding/intensive for developers; pair programming uses more staff time; needs a constantly available customer.
- *Use when:* high-quality, low-risk code is needed and the team can work closely with the customer.

**Spiral** — combines iterative development with a strong focus on **risk analysis**. Each loop of the spiral covers planning, risk assessment, development and evaluation; risky elements are tackled early.
- *Pros:* manages risk well; good for large, expensive, high-risk projects; prototypes refine requirements.
- *Cons:* expensive and complex; risk analysis needs expertise; overkill for small projects.
- *Use when:* the project is large, costly and high-risk.

**Rapid application development (RAD)** — builds **prototypes** quickly and refines them through user feedback in repeated cycles; favours fast, partially working models over heavy upfront planning.
- *Pros:* fast delivery; strong user involvement; good where requirements are vague.
- *Cons:* needs skilled developers and committed users; less suitable for very large systems; can produce a less robust system.
- *Use when:* a system is needed quickly and user feedback can shape it.

| Methodology | Key idea | Best when… |
|---|---|---|
| Waterfall | Linear, one stage at a time | Requirements fixed and clear |
| Agile | Iterative increments, client feedback | Requirements likely to change |
| Extreme programming | Agile + pair programming + constant testing | High code quality essential |
| Spiral | Iterative + risk analysis | Large, expensive, high-risk project |
| RAD | Rapid prototyping, user feedback | Need it fast / vague requirements |

> 💡 **A\* extension**
> In an exam scenario, *justify* your choice against the brief: a fixed-scope, regulated system (e.g. payroll with legal rules) suits **waterfall**; a start-up app whose features will evolve suits **agile**; a safety-critical, multi-million-pound system suits **spiral**; a quick internal tool to be shaped by users suits **RAD**. Always reference the specific clues in the question (budget, deadline, how clear the requirements are, how risky the project is, client availability).

### Stages of the lifecycle

1. **Analysis** — investigate the current system and gather/define **requirements**; produce a requirements specification. (Includes a feasibility study.)
2. **Design** — plan *how* the system will meet the requirements: data structures, algorithms, interface, file/database design, inputs/outputs.
3. **Development / implementation** — write (code) the program from the design, usually in modules.
4. **Testing** — check the program against the requirements; find and fix errors using test data (normal, boundary, erroneous), e.g. alpha and beta testing.
5. **Evaluation** — assess the finished system against the original requirements: is it effective, usable, robust, and maintainable? Identify improvements.

## 1.2.4 Types of programming language

### The need for different paradigms

A **programming paradigm** is a style or way of approaching how a program is written. Different problems are best solved with different paradigms, so no single language suits every task. For example, low-level **assembly** gives fine control of hardware (drivers, embedded systems), **procedural** code suits step-by-step tasks, and **object-oriented** code suits large systems modelled on real-world objects. Choosing the right paradigm makes a program easier to write, maintain and reuse.

### Procedural languages

A **procedural language** executes a sequence of instructions/statements in order, using **sequence, selection and iteration**, and organises code into reusable **procedures/subroutines (functions)**. The programmer specifies *how* to solve the problem step by step. Examples: Python, C, Pascal, VB.

### Assembly language and the Little Man Computer (LMC)

**Assembly language** is a **low-level language** that uses **mnemonics** (short codes like `ADD`, `STA`) instead of binary machine code. It is generally **one mnemonic = one machine-code instruction**, and is specific to a processor's instruction set. It is used where speed, small size or direct hardware control matter (device drivers, embedded systems).

The **Little Man Computer (LMC)** is a simple teaching model of a CPU with a small instruction set, used to learn assembly and the fetch–decode–execute cycle.

**Common LMC instructions:**

| Mnemonic | Meaning |
|---|---|
| `INP` | Take user input into the accumulator |
| `OUT` | Output the value in the accumulator |
| `STA xx` | Store accumulator value into mailbox/address xx |
| `LDA xx` | Load value from mailbox xx into accumulator |
| `ADD xx` | Add value at xx to the accumulator |
| `SUB xx` | Subtract value at xx from the accumulator |
| `BRA xx` | Branch always (unconditional jump) to xx |
| `BRZ xx` | Branch to xx if accumulator = 0 |
| `BRP xx` | Branch to xx if accumulator ≥ 0 (positive) |
| `DAT` | Define a data label / memory location |
| `HLT` | Stop the program |

### Modes of addressing memory

The **operand** of an instruction can be interpreted in different ways. These are **addressing modes**:

| Mode | The operand is… | Example meaning |
|---|---|---|
| **Immediate** | the **actual value** to use | `ADD #5` → add the literal value 5 |
| **Direct** | the **address** that holds the value | `ADD 20` → add the contents of address 20 |
| **Indirect** | the address of a location that **holds the address** of the value | `ADD (20)` → address 20 holds another address; use the value *there* |
| **Indexed** | a **base address + the contents of an index register** | used to step through arrays/lists |

> 💡 **A\* extension**
> Indirect addressing allows access to more memory than the operand field alone could address, and is used for **pointers**. Indexed addressing is ideal for arrays: keep the array's start address fixed and increment the index register to move through elements in a loop. Be able to *trace* what value is finally added/loaded for each mode given a small memory table — that is a classic exam trap.

### Object-oriented (OO) concepts

| Term | Definition |
|---|---|
| **Class** | A **template/blueprint** that defines the attributes and methods that objects of that type will have |
| **Object** | A specific **instance** of a class |
| **Instantiation** | The process of **creating an object** from a class |
| **Attribute / property** | A **variable** belonging to a class/object (its data/state) |
| **Method** | A **subroutine/function** belonging to a class (its behaviour) |
| **Encapsulation** | Bundling attributes and methods together and **hiding the data** (private attributes accessed only through methods — *information hiding*) |
| **Inheritance** | A **child (sub) class** inherits the attributes and methods of a **parent (super) class**, and can add or override its own |
| **Polymorphism** | The ability of methods to **behave differently** depending on the object — e.g. overriding an inherited method so a `Dog` and a `Cat` respond to `makeSound()` differently |

**Class diagrams** show classes as boxes split into three: the **class name**, the **attributes**, and the **methods**. Visibility is shown with `+` (public) or `-` (private). Inheritance is shown with an arrow pointing from the child class to the parent class.

```
        +----------------------+
        |       Animal         |   <-- parent / super class
        +----------------------+
        | - name : string      |   <-- attributes (- = private)
        | - age : integer      |
        +----------------------+
        | + makeSound()        |   <-- methods (+ = public)
        | + getName()          |
        +----------------------+
                  ^
                  |  (inheritance arrow points to parent)
        +----------------------+
        |        Dog           |   <-- child / sub class
        +----------------------+
        | - breed : string     |
        +----------------------+
        | + makeSound()        |   <-- overridden (polymorphism)
        +----------------------+
```

> 💡 **A\* extension**
> Encapsulation makes code more **maintainable and robust**: because data is private and accessed only through methods (getters/setters), the internal implementation can change without breaking other code, and invalid values can be rejected. Inheritance avoids duplicating code (**DRY** — *don't repeat yourself*) and models "is-a" relationships. Polymorphism lets you write code that works on the parent type but calls the correct overridden method at run time. In code, use the `inherits`/`extends` keyword and call the parent constructor with `super`.

## Worked examples

### 1. LMC program walkthrough

This program reads two numbers and outputs the larger one.

```
        INP            // input first number
        STA first      // store it in 'first'
        INP            // input second number
        STA second     // store it in 'second'
        SUB first      // accumulator = second - first
        BRP secbig     // if (second - first) >= 0, second is bigger (or equal)
        LDA first      // otherwise first is bigger -> load first
        OUT
        HLT
secbig  LDA second     // load and output second
        OUT
        HLT
first   DAT            // data location for first number
second  DAT            // data location for second number
```

**Trace with inputs 4 then 9:**
1. `INP` → accumulator = 4; `STA first` → first = 4.
2. `INP` → accumulator = 9; `STA second` → second = 9.
3. `SUB first` → accumulator = 9 − 4 = 5.
4. `BRP secbig` → 5 ≥ 0, so branch to `secbig`.
5. `LDA second` → accumulator = 9; `OUT` → outputs **9**; `HLT`.

### 2. Addressing-mode example

Suppose the instruction operand is `20`, and memory looks like this:

| Address | Contents |
|---|---|
| 20 | 35 |
| 35 | 99 |

For `LDA 20`:
- **Immediate** → loads the value **20** (the operand itself).
- **Direct** → loads the contents of address 20 → **35**.
- **Indirect** → address 20 holds 35, so go to address 35 → loads **99**.
- **Indexed** (if index register = 0) → loads contents of address (20 + 0) = address 20 → **35**; if index register = 15, loads address 35 → **99**.

### 3. Methodology-choice scenario

**Scenario:** "A hospital commissions a new patient-monitoring system that must obey strict safety regulations and is expected to cost several million pounds. Recommend a development methodology and justify your choice."

**Model answer:** The **spiral** model is most appropriate. The project is large, expensive and high-risk (patient safety), and the spiral model places **risk analysis** at the centre of each iteration so dangerous elements are identified and prototyped early. Each loop refines the requirements through prototyping, which suits a complex system, and the repeated evaluation reduces the chance of an expensive failure late on. Waterfall would be too inflexible for such a complex system, and pure agile/RAD would not give the rigorous risk management and documentation needed for a safety-critical, regulated product.

## Common exam mistakes & examiner tips

- **Paging vs segmentation:** the *only* reliable discriminator is fixed-size (paging) vs variable-size/logical (segmentation). State it explicitly.
- **Virtual memory:** don't say it "adds RAM". It uses *secondary storage* as if it were RAM — it is much slower than real RAM. Mention thrashing for the extra mark.
- **Interrupts:** the CPU checks for interrupts at the **end of each cycle**, and saves registers on a **stack** so the original job can resume. Forgetting the stack loses marks.
- **Compiler vs interpreter:** the key differences are *when* translation happens and *whether an executable is produced* — quote both. Don't say a compiler is "always faster"; the *resulting program* runs faster, but compiling takes time upfront.
- **Stages of compilation:** keep them in order and don't muddle lexical (tokens) with syntax (grammar). Errors of grammar are found in *syntax* analysis.
- **Methodology questions are AO2/AO3** — you must *apply and justify* against the scenario's clues, not just describe the methodology. A pure description caps your marks.
- **Addressing modes:** read carefully whether the operand is a value or an address, and trace through any memory table given. Indirect = "address of an address".
- **OO terms:** keep class (template) and object (instance) distinct; encapsulation is *data hiding*, inheritance is *reuse*, polymorphism is *same method name, different behaviour*. Give an example for full marks.
- **Utilities vs applications:** a utility maintains the *computer*; an application does the *user's task*. Antivirus is a utility, not an application.
- **BIOS** loads the OS; it is *not* the OS itself.

## Exam-style questions

**Q1 (AO1, 2 marks).** State what is meant by *virtual memory* and give one reason it is used.

<details><summary>Mark scheme</summary>

- Using secondary storage / part of the hard disk as if it were RAM (1).
- To allow programs larger than physical RAM to run / to allow more programs to be open at once than would fit in RAM (1).
</details>

**Q2 (AO1, 4 marks).** Describe the four main stages of compilation.

<details><summary>Mark scheme</summary>

1 mark each (max 4):
- **Lexical analysis** — source code broken into tokens; whitespace/comments removed; symbol table created.
- **Syntax analysis (parsing)** — tokens checked against the grammar/rules of the language; syntax errors reported; parse tree built.
- **Code generation** — machine/object code produced from the parse tree and symbol table.
- **Optimisation** — code refined to run faster / use less memory while giving the same result.
</details>

**Q3 (AO1, 6 marks).** Compare the use of a **compiler** with an **interpreter** when developing and distributing software.

<details><summary>Mark scheme</summary>

Up to 6 marks for paired comparison points, e.g.:
- A compiler translates the whole program at once into an executable; an interpreter translates and runs one line at a time (1–2).
- A compiler produces a standalone executable that can be distributed without source code, protecting intellectual property; an interpreter needs the source and an interpreter present to run (1–2).
- Compiled programs run faster as they are already machine code; interpreted programs are slower as they are translated each run (1).
- An interpreter is better during development/debugging because it reports errors line by line; a compiler reports errors only after the whole compilation (1).
- Interpreted code is more portable (any platform with an interpreter); compiled executables are platform-specific (1).

Mark band: a top-band answer makes balanced, correct comparisons relevant to *both* developing and distributing.
</details>

**Q4 (AO2, 4 marks).** A washing machine and a passenger aircraft each contain an operating system. Identify the most suitable type of OS for each and justify your choice.

<details><summary>Mark scheme</summary>

Washing machine:
- **Embedded OS** (1) — it is a dedicated device with limited resources; the OS is small, efficient and stored in ROM to control one specific task reliably (1).

Aircraft:
- **Real-time OS** (1) — control actions (e.g. responding to sensors) must happen within strict deadlines; an RTOS guarantees a response within a set time, which is essential for safety (1).
</details>

**Q5 (AO2, 6 marks).** Memory contains: address 30 holds the value 12; address 12 holds the value 7. The instruction is `LDA 30`. State what value is loaded into the accumulator under (a) immediate, (b) direct, (c) indirect addressing, and explain each.

<details><summary>Mark scheme</summary>

(a) **Immediate = 30** — the operand itself is the value loaded (1 + 1 explanation).
(b) **Direct = 12** — the operand is an address; the contents of address 30 (= 12) are loaded (1 + 1).
(c) **Indirect = 7** — address 30 holds 12, which is itself an address; the contents of address 12 (= 7) are loaded (1 + 1).
</details>

**Q6 (AO3, 9 marks).** A small charity needs a new membership database. The exact features are not yet clear and the charity's staff want to be closely involved and to see progress early. Discuss which software development methodology should be used, and evaluate your recommendation against alternatives.

<details><summary>Mark scheme</summary>

Levelled response (indicative content):

Recommendation — **Agile** (or **RAD**):
- Requirements are unclear/likely to change, which suits an **iterative** approach rather than a fixed linear plan.
- Staff want close involvement and early sight of progress — agile delivers **working increments** with continuous client feedback; **RAD** uses rapid prototypes shaped by users.
- Changes can be incorporated cheaply between iterations.

Evaluation of alternatives:
- **Waterfall** rejected — it is linear and inflexible, the client sees the product only at the end, and it needs fixed requirements that the charity does not have.
- **Spiral** rejected — its heavy risk analysis and cost are overkill for a small, low-risk charity database.
- **Extreme programming** could give high quality but its intensity/pair programming may be more than a small project needs, though its constant customer involvement fits.

Mark bands:
- **Level 3 (7–9):** clear, justified recommendation that applies *specific clues from the scenario* (unclear requirements, wanting early progress and involvement) and evaluates alternatives, reaching a reasoned conclusion.
- **Level 2 (4–6):** sound recommendation with some application/justification; limited evaluation of alternatives.
- **Level 1 (1–3):** mainly description of methodologies with little application to the scenario.
</details>

**Q7 (AO1/AO2, 5 marks).** Explain how the processor handles an interrupt, referring to the interrupt service routine.

<details><summary>Mark scheme</summary>

1 mark each (max 5):
- At the end of the current fetch–decode–execute cycle the CPU checks for interrupts.
- If an interrupt of sufficient priority is present, the current instruction is completed.
- The contents of the registers (PC, accumulator, etc.) are saved/pushed onto the **stack**.
- The CPU loads and runs the appropriate **interrupt service routine** to deal with the interrupt.
- When the ISR finishes, the saved register contents are restored/popped from the stack and the interrupted task resumes.
- (Also credit: lower-priority interrupts are queued/held until higher-priority handling completes.)
</details>

**Q8 (AO1, 4 marks).** Define *encapsulation* and *inheritance* in object-oriented programming, giving one benefit of each.

<details><summary>Mark scheme</summary>

- **Encapsulation** — bundling attributes and methods together and hiding the data (private attributes accessed via methods) (1). Benefit: more robust/maintainable; data protected from invalid changes / implementation can change without breaking other code (1).
- **Inheritance** — a subclass acquires the attributes and methods of a superclass (1). Benefit: avoids duplicating code / promotes reuse (1).
</details>
