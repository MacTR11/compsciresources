## 1.1.1 Structure and function of the processor

The **processor (CPU — Central Processing Unit)** is the component that fetches, decodes and executes the instructions of a program. It is built from two key internal components plus a set of registers.

### The ALU and Control Unit

- **Arithmetic Logic Unit (ALU):** carries out all **arithmetic** operations (e.g. add, subtract, multiply) and all **logical** operations (e.g. AND, OR, NOT, comparisons) and **shifts**. The result of a calculation is normally placed in the Accumulator.
- **Control Unit (CU):** **manages and coordinates the activities of the CPU and the rest of the computer.** It decodes each instruction, generates control signals to direct the other components, manages the flow of data along the buses, and synchronises operations using the **system clock**.

### Registers

A **register** is a **small, extremely fast block of memory inside the CPU** used to temporarily hold data during processing. The five you must know:

| Register | Full name | Function |
|----------|-----------|----------|
| **PC** | Program Counter | Holds the **address of the next instruction** to be fetched. Incremented during each cycle. |
| **MAR** | Memory Address Register | Holds the **address** of the memory location currently being read from or written to. |
| **MDR** | Memory Data Register | Holds the **data or instruction** that has just been fetched from, or is about to be written to, memory. |
| **CIR** | Current Instruction Register | Holds the **current instruction** while it is being decoded and executed. |
| **ACC** | Accumulator | Stores the **immediate results** of calculations carried out by the ALU. |

> 💡 **A\* extension:** The MAR/MDR pairing is the bridge between CPU and memory. The MAR's width (number of bits) determines how many memory locations can be addressed (an *n*-bit address bus → 2ⁿ addressable locations), and the MDR's width relates to how much data moves per transfer (the data bus width). A wider address bus increases the maximum addressable memory; a wider data bus increases throughput per fetch.

### Buses

A **bus** is a set of parallel wires that carries data between components. There are three system buses:

| Bus | Direction | Carries |
|-----|-----------|---------|
| **Address bus** | **Unidirectional** (CPU → memory) | The **memory address** of the location to read/write. Width determines max addressable memory (2ⁿ). |
| **Data bus** | **Bidirectional** | The actual **data/instructions** being transferred between CPU and memory. |
| **Control bus** | **Bidirectional** | **Control signals** (e.g. read/write, clock, interrupt request, bus request) to coordinate activity. |

### The Fetch–Decode–Execute cycle

Every instruction is processed by repeating this cycle. Know the **register transfers** at each stage.

**FETCH** — get the next instruction from memory:
1. The **address in the PC** is copied to the **MAR**.
2. The instruction at that address is fetched along the **data bus** into the **MDR** (a *read* signal goes along the control bus).
3. The **PC is incremented** (by 1) so it points to the next instruction.
4. The contents of the **MDR are copied into the CIR**.

**DECODE** — the **CU decodes** the instruction in the CIR, splitting it into the **opcode** (the operation) and the **operand** (the data/address it acts on).

**EXECUTE** — the decoded instruction is carried out. This may involve the **ALU** performing a calculation (result → **ACC**), loading/storing data in memory, or changing the PC (a jump). The cycle then repeats.

```text
FETCH:   [PC] -> MAR
         (memory read via address bus; control bus = READ)
         data/instruction from memory -> MDR (via data bus)
         PC := PC + 1
         [MDR] -> CIR
DECODE:  CU splits CIR into opcode + operand
EXECUTE: action carried out (ALU op -> ACC, memory access, or jump)
         -> repeat
```

> 💡 **A\* extension:** A **jump/branch** instruction works by writing a new address directly into the **PC** during the execute phase, so the next fetch comes from a different place — this is how loops and selection are implemented in machine code. This also explains why pipelining suffers from *branch hazards*: instructions fetched ahead of a taken branch must be discarded.

### Factors affecting CPU performance

**Clock speed** — The **clock** generates regular electrical pulses that synchronise the CPU's operations; **clock speed** is measured in **hertz (Hz)** (e.g. 3.5 GHz = 3.5 billion cycles per second). A higher clock speed means more FDE cycles per second → faster processing. Limits: heat dissipation and power consumption rise sharply with clock speed (limiting *overclocking*).

**Number of cores** — A **core** is an independent processing unit capable of running its own FDE cycle. A **multicore** CPU (dual-core, quad-core, etc.) can execute **multiple instructions simultaneously**, increasing throughput — *but only if the software is written to use multiple cores* and the task can be split up. Doubling cores rarely doubles real-world performance.

**Cache** — **Cache is a small amount of very fast memory located on or very close to the CPU** that stores frequently/recently used instructions and data, so the CPU does not have to fetch them from slower main memory (RAM) each time. More/faster cache → fewer slow RAM accesses → faster performance.

**Cache levels:**

| Level | Speed | Size | Location |
|-------|-------|------|----------|
| **L1** | Fastest | Smallest | On the core itself |
| **L2** | Slower than L1 | Larger than L1 | On/near the core |
| **L3** | Slowest of the three (still faster than RAM) | Largest | Often shared between cores |

The CPU checks L1 first, then L2, then L3, then RAM. A **cache miss** (data not in cache) forces a slower fetch.

**Pipelining** — **Pipelining splits instruction processing into stages (e.g. fetch, decode, execute) so that while one instruction is being executed, the next can be decoded and a third can be fetched — at the same time.** This keeps the parts of the CPU busy and increases instruction throughput.

> 💡 **A\* extension:** Pipelining is limited by **hazards**: a *branch/control hazard* (a jump invalidates the pre-fetched instructions, which must be flushed) and a *data hazard* (an instruction needs the result of one still in the pipeline). Techniques such as **branch prediction** and **out-of-order execution** mitigate these. There is a trade-off between high clock speed and deep pipelines versus the penalty when the pipeline must be flushed.

## 1.1.2 Types of processor

### Von Neumann vs Harvard architecture

| Feature | **Von Neumann** | **Harvard** |
|---------|-----------------|-------------|
| Memory | **One** shared memory for both **data and instructions** | **Separate** memories for data and instructions |
| Buses | Shared bus | Separate buses for data and instructions |
| Access | Cannot fetch an instruction and read/write data at the same time (the **Von Neumann bottleneck**) | Can fetch an instruction *and* access data simultaneously |
| Cost/complexity | Simpler, cheaper, more flexible | More complex, more expensive |
| Typical use | General-purpose computers (PCs, laptops) | Embedded systems, microcontrollers, DSPs |

**Von Neumann architecture:** a single store holds both program instructions and data, accessed over a shared bus. **Harvard architecture:** instructions and data are stored in physically separate memories with separate buses, allowing them to be accessed at the same time.

> 💡 **A\* extension:** Modern CPUs are often a **hybrid (modified Harvard)**: main memory is unified (Von Neumann) but the on-chip **L1 cache is split** into a separate instruction cache and data cache, giving Harvard-style simultaneous access at cache level while keeping a simple unified main memory.

### CISC vs RISC

| Feature | **CISC** (Complex Instruction Set Computer) | **RISC** (Reduced Instruction Set Computer) |
|---------|---------------------------------------------|---------------------------------------------|
| Instructions | Large number of complex instructions | Small number of simple instructions |
| Instruction length | Variable; one instruction may take several cycles | Fixed length; aim for one instruction per cycle |
| Hardware vs software | Complexity in **hardware** | Complexity in **software/compiler** (more instructions per task) |
| Pipelining | Harder (variable-length instructions) | Easier (uniform, single-cycle instructions) |
| Power use | Generally higher | Lower → good for battery devices |
| Typical use | Traditional desktop x86 CPUs | Mobile/embedded (e.g. ARM) |

> 💡 **A\* extension:** RISC's simple, uniform instructions make pipelining and parallelism much easier and reduce power consumption — which is why RISC (ARM) dominates smartphones, tablets and increasingly laptops. CISC keeps complexity in hardware so each program needs fewer instructions and less compiler effort. In practice modern x86 chips translate CISC instructions into RISC-like *micro-ops* internally.

### GPUs (Graphics Processing Units)

A **GPU is a processor specialised for performing the same operation on large blocks of data in parallel.** It contains thousands of small cores optimised for **highly parallel** floating-point work.

- **Graphics use:** rendering images, transforming and shading the many pixels/vertices of 3D scenes simultaneously.
- **Non-graphics uses (GPGPU):** any task that applies the same calculation across large data sets in parallel — e.g. **machine learning / training neural networks**, **scientific/numerical simulations**, **image and video processing**, and **cryptocurrency mining / brute-force hashing**.

> 💡 **A\* extension:** GPUs excel at **data parallelism**: identical, independent operations on many data items. They are poor at heavily *sequential*, branch-heavy logic with dependencies — that work suits the few powerful, high-clock cores of a CPU. This is why CPU and GPU are complementary.

### Multicore and parallel systems

- **Multicore:** multiple cores on a single chip, each able to process instructions independently and at the same time.
- **Parallel processing:** **carrying out multiple computations simultaneously**, splitting a task across several processors/cores to complete it faster.

**Why some tasks don't parallelise well:**
- **Data dependency:** when one step needs the result of a previous step, it must wait — it cannot run in parallel (inherently sequential work).
- **Overheads:** splitting the task, coordinating cores and recombining results adds management overhead.
- **Unsuitable problems:** some algorithms are fundamentally sequential and cannot be divided.

> 💡 **A\* extension:** This is captured by the **serial fraction** idea: the maximum speed-up is limited by the proportion of the task that *must* run serially. If even 10% of a task is sequential, the best possible speed-up is ×10 no matter how many cores you add. The serial fraction, plus synchronisation overhead, is why "twice the cores" almost never means "twice as fast".

## 1.1.3 Input, output and storage devices

### Choosing input and output devices

There is no single "correct" device — you must **justify a choice against the scenario's requirements**. Consider the **environment**, **user needs/accessibility**, **accuracy**, **speed**, **cost** and **durability**.

| Device | Type | How it works (briefly) | Good for |
|--------|------|------------------------|----------|
| Optical mouse | Input | LED + sensor tracks surface movement | General pointing |
| Keyboard | Input | Key press completes a circuit → scancode | Text entry |
| Touchscreen (capacitive) | Input/Output | Detects change in electrical charge where a finger touches | Phones, kiosks, tablets |
| Barcode scanner | Input | Light reflected off bars/spaces read as a pattern → number | Stock/checkout |
| Microphone | Input | Converts sound waves to an electrical/digital signal | Voice input, calls |
| Sensor | Input | Measures a physical quantity (temp, light, motion) | Control/embedded systems |
| Monitor (LCD/LED) | Output | Pixels emit/filter light to form an image | Visual display |
| Laser printer | Output | Laser charges a drum; toner sticks and is fused to paper | Fast, high-volume, sharp text |
| Inkjet printer | Output | Sprays tiny droplets of ink onto paper | Cheap colour, photos, low volume |
| 3D printer | Output | Builds an object layer by layer from a model (e.g. melted filament) | Prototyping, manufacturing |
| Speakers | Output | Convert an electrical signal into sound waves | Audio output |
| Actuator | Output | Converts a signal into physical movement | Robotics/control |

> 💡 **A\* extension:** Strong answers *trade off* options against the **specific** scenario. Example: for a busy office printing thousands of black-and-white documents, a **laser printer** wins on speed and cost-per-page; for a home user printing occasional colour photos, an **inkjet** wins on lower upfront cost and photo quality. Always tie the device characteristic back to a stated requirement.

### Storage: magnetic, flash and optical

Storage here means **secondary storage** — **non-volatile** storage that retains data without power (unlike RAM).

**Magnetic storage (e.g. HDD, magnetic tape):** data is stored as **magnetised patterns (polarity)** on a spinning platter (HDD) or tape; a read/write head detects/changes magnetisation.

| Pros | Cons |
|------|------|
| Very large capacity, low cost per GB | Slower than flash (moving parts) |
| Long-established, reliable for archiving (tape) | Fragile — moving parts damaged by knocks |
| | Higher power use, noise, heat |

**Flash / solid-state storage (e.g. SSD, USB memory stick, SD card):** a type of EEPROM that stores data by **trapping electrons in transistors (memory cells)** — **no moving parts**.

| Pros | Cons |
|------|------|
| Very fast access; silent; low power | More expensive per GB than magnetic |
| Durable, shock-resistant, portable | Finite number of write cycles (wears out) |

**Optical storage (e.g. CD, DVD, Blu-ray):** a **laser** reads/writes data by detecting **pits and lands** burned into a reflective surface; the laser detects changes in reflected light.

| Pros | Cons |
|------|------|
| Very cheap, portable | Low capacity vs HDD/SSD |
| Good for distributing media; long shelf life | Slow; easily scratched; many devices no longer have drives |

> 💡 **A\* extension:** Match storage to the access pattern. **Magnetic tape** is *serial* access (must wind through sequentially) — terrible for random access but excellent and cheap for **backup/archiving**. HDDs and SSDs are **direct (random) access**. Flash wear is managed by **wear levelling**, which spreads writes evenly across cells to extend lifespan.

### RAM vs ROM

| Feature | **RAM** (Random Access Memory) | **ROM** (Read Only Memory) |
|---------|-------------------------------|----------------------------|
| Volatility | **Volatile** — loses contents when power off | **Non-volatile** — keeps contents when power off |
| Read/write | Read **and** write | Normally **read only** |
| Contents | Currently running programs and data | Boot/firmware instructions (e.g. the bootstrap/BIOS) |
| Speed/use | Fast working memory the CPU uses while running | Permanent start-up instructions |

**RAM:** fast, volatile main memory holding the programs and data currently in use. **ROM:** non-volatile memory holding permanent instructions such as the bootstrap loader that starts the computer.

### Virtual storage / virtual memory

**Virtual memory:** when **RAM is full**, the operating system uses a **portion of secondary storage (a "swap file"/paging file) as if it were extra RAM**, moving pages of data not currently needed out of RAM and back in when required.

- **Pro:** lets you run more/larger programs than physical RAM alone would allow.
- **Con:** secondary storage is far slower than RAM, so excessive paging causes **"disk thrashing"** and slows the system.

> 💡 **A\* extension:** Note the distinction between **virtual memory** (using disk as overflow for RAM, above) and broader **virtual/cloud storage** (storing data on remote servers accessed over a network — benefits: access anywhere, scalability, off-site backup; drawbacks: needs internet, ongoing cost, security/privacy concerns). Read the question carefully to see which is meant.

## Worked examples

### Worked example 1 — Full Fetch–Decode–Execute trace

> **Q:** Describe the register transfers that take place during the **fetch** phase of the FDE cycle. *(4 marks, AO1)*

**Model answer:**
1. The address held in the **PC** is copied into the **MAR**.
2. A **read** signal is sent on the **control bus**; the instruction at that address travels back along the **data bus** into the **MDR**.
3. The **PC is incremented** so it points to the next instruction.
4. The contents of the **MDR** are copied into the **CIR**, ready for decoding.

### Worked example 2 — Performance comparison scenario

> **Q:** Two CPUs are offered. CPU A: 4.0 GHz, **dual-core**, 4 MB cache. CPU B: 3.2 GHz, **eight-core**, 16 MB cache. A user mainly runs **video editing/rendering** software designed for multiple cores. Recommend a CPU and justify your choice. *(6 marks, AO3)*

**Model answer:**
- **CPU B is the better choice for this user.** Although CPU A has a higher **clock speed** (4.0 GHz vs 3.2 GHz), so each individual core completes more cycles per second, video rendering is a **highly parallelisable** task that can be split across many cores.
- CPU B has **eight cores** versus two, so it can process **far more instructions simultaneously**, which is the dominant factor for rendering throughput, more than outweighing the lower clock speed.
- CPU B also has **four times the cache** (16 MB vs 4 MB), so more frequently-used instructions/data are held in fast memory close to the CPU, reducing slow RAM fetches (cache misses) during large rendering jobs.
- *Caveat:* the software must actually be written to use all cores; for purely single-threaded tasks CPU A's higher clock speed could win. Given the stated multi-core workload, **CPU B is recommended.**

## Common exam mistakes & examiner tips

- **Don't confuse MAR and MDR.** MAR = **address**, MDR = **data**. Mixing these loses easy marks.
- In the fetch trace, **remember to increment the PC** and to copy MDR → CIR — both are commonly forgotten.
- The **address bus is unidirectional**; the data and control buses are **bidirectional**. State this correctly.
- "More cores = faster" is **only true if the task can be parallelised** and the software supports it — always add this caveat.
- Don't say cache is "part of RAM"; it is **separate, faster memory on/near the CPU**.
- Distinguish **virtual memory** (disk used as overflow RAM) from **cloud/virtual storage** (remote servers).
- For "choose a device" questions, **link each device characteristic to the scenario** — generic pros/cons without application caps your marks at AO1.
- Flash (SSD) is **non-volatile** — don't confuse it with RAM just because both have "no moving parts" feel; RAM is volatile.
- Define RISC/CISC by **instruction set and where complexity lies**, not vaguely as "faster/slower".
- When comparing, **actually compare** (use "whereas"/"compared to") rather than listing two separate descriptions.

**Q3 (AO1, 4 marks).** Explain how **cache** improves CPU performance and describe the difference between **L1** and **L2/L3** cache.

<details><summary>Mark scheme</summary>

- Cache is small, very fast memory on/near the CPU (1) that stores frequently/recently used instructions and data (1).
- The CPU can fetch these without accessing slower main memory (RAM), reducing waiting time (1).
- L1 is smallest and fastest, on the core; L2 then L3 are progressively larger but slower (L3 often shared between cores) (1).
</details>

**Q4 (AO2, 6 marks).** Compare **Von Neumann** and **Harvard** architectures, giving a suitable use for each.

<details><summary>Mark scheme (max 6)</summary>

- Von Neumann uses **one shared memory** for data and instructions, accessed over a shared bus (1); Harvard uses **separate memories and buses** for data and instructions (1).
- Harvard can fetch an instruction and access data **simultaneously**, whereas Von Neumann cannot — the Von Neumann bottleneck (1).
- Von Neumann is simpler/cheaper/more flexible (1); Harvard is more complex/expensive but faster for its purpose (1).
- Suitable use of Von Neumann: general-purpose computers/PCs (1). Suitable use of Harvard: embedded systems/microcontrollers/DSPs (1).
</details>

**Q5 (AO2, 4 marks).** A scientist needs to **train large neural networks** that apply the same calculation across huge data sets. Explain why a **GPU** is well suited to this task.

<details><summary>Mark scheme</summary>

- A GPU has **many (thousands of) cores** designed for parallel processing (1).
- Neural network training applies the **same operation to large blocks of data** (data parallelism) (1).
- These independent calculations can be performed **simultaneously** across the cores, greatly speeding up training (1).
- This is faster than a CPU, which has few cores optimised for sequential tasks (1).
</details>

**Q6 (AO3, 9 marks).** A company is choosing **secondary storage** for two needs: (a) the **internal drive of a laptop** used by sales staff who travel, and (b) **long-term backup/archiving** of large volumes of historical data. Recommend and justify a suitable storage technology for each, evaluating the trade-offs.

<details><summary>Mark scheme (levelled, max 9)</summary>

Indicative content — credit a reasoned, applied evaluation:

**(a) Laptop drive — recommend flash/SSD:**
- No moving parts → durable and **shock-resistant**, important for a travelling user (1).
- **Fast** access and **low power** (better battery life), **silent**, lightweight/portable (1).
- Trade-off: more expensive per GB and finite write cycles, but capacities are sufficient for a laptop (1).

**(b) Backup/archiving — recommend magnetic (HDD or tape):**
- **Very large capacity at low cost per GB**, ideal for large historical volumes (1).
- **Tape** in particular is cheap and reliable for long-term archive, though serial access makes random retrieval slow (1).
- Trade-off: slower and has moving parts/fragility, but for infrequently accessed backups speed matters less (1).

**Evaluation / comparison (for higher levels):**
- Explicitly weighs cost vs speed vs durability vs access pattern for **each** scenario (1).
- Considers optical (cheap, portable, but too low capacity for large archives) and rejects with reason (1).
- Reaches a clear, justified recommendation for both (1).

*Level descriptors:* L3 (7–9) sustained, well-applied evaluation with justified choices for both; L2 (4–6) some application and comparison; L1 (1–3) basic descriptions with limited application.
</details>

**Q7 (AO1/AO2, 3 marks).** Explain **why doubling the number of cores in a CPU does not usually double a program's speed.**

<details><summary>Mark scheme</summary>

- The program must be **written to use multiple cores**; single-threaded code cannot use the extra cores (1).
- Some tasks have **data dependencies** / are inherently sequential, so steps must wait for earlier results (1).
- There is **overhead** in splitting work, coordinating cores and recombining results (the serial fraction limits the speed-up) (1).
</details>

**Q8 (AO1, 4 marks).** Compare **RAM** and **ROM**.

<details><summary>Mark scheme</summary>

- RAM is **volatile** (loses contents when power off); ROM is **non-volatile** (retains contents) (1).
- RAM can be **read and written**; ROM is normally **read only** (1).
- RAM holds **programs/data currently in use**; ROM holds **permanent start-up/bootstrap instructions** (1).
- RAM is used as fast working memory while running; ROM provides instructions needed at boot (1).
</details>
