# Component 03/04 — Programming Project (NEA): How to Score Full Marks

The Programming Project is worth **70 marks (20% of the A Level)** and is almost entirely **AO3** (design, program, evaluate). It is the most *controllable* 20% on the course: there is no exam-day risk, and the mark scheme tells you exactly what to evidence. Treat it as bankable A\* marks.

> ⚠️ This is a coursework (non-exam assessment) guide to maximise marks **through your own work**. All code and writing must be your own; follow your centre's and OCR's rules on authenticity and acknowledge any external sources/libraries. Always work from the **current** OCR marking criteria your teacher gives you.

---

## The four assessed stages (and their weighting)

The project is marked across four stages. Roughly:

| Stage | What it is | Approx. marks |
|-------|-----------|---------------|
| **Analysis** | Investigate the problem, identify stakeholders, set measurable requirements/success criteria | ~10 |
| **Design** | Plan the solution: structure, algorithms, data, UI, and a test plan | ~15 |
| **Development & testing** | Build it iteratively with evidence and testing throughout | ~25 |
| **Evaluation** | Test against success criteria, evaluate, and propose improvements | ~20 |

(Use the exact mark breakdown on your centre's current OCR criteria — proportions above are indicative.)

---

## Choosing a project that scores well

The biggest single decision. To access the **top band**, the problem must allow **complex, fully-investigated** solutions with **algorithmic content** — not just CRUD forms.

✅ **Good** signs:
- A **real stakeholder** you can interview (teacher, club, small business, family member).
- Genuine **computational complexity**: searching/sorting, pathfinding, simulation, rules/logic, optimisation, AI opponent, file/DB handling, data analysis.
- Naturally **modular** so you can develop and test in iterations.

❌ **Avoid**:
- A simple website/quiz with no algorithmic depth.
- A problem so big you can't finish or evaluate it.
- A clone with no real user or measurable requirements.

> 💡 **A\* signal:** examiners reward solutions where the *complexity is evidenced in the design and code* (e.g. you implemented and justified a custom algorithm/data structure), not bolted-on claims.

## Stage 1 — Analysis (~10 marks)

**Goal:** prove the problem is real, well-understood and solvable computationally.

- **Identify the problem** and why it suits a computational solution (link to 2.2.2 features that make a problem solvable).
- **Identify the stakeholder(s)** specifically (named, with their needs) — generic "users" loses marks.
- **Research:** interview the stakeholder; look at existing/similar solutions; record findings.
- **Requirements & measurable success criteria** — each criterion must be **testable** later (e.g. "the user can search bookings by date in < 2 s"). This list is what your evaluation will measure against, so make it thorough and specific.

**Top-band tip:** show abstraction and decomposition explicitly; justify every requirement against research/stakeholder need.

---

## Stage 2 — Design (~15 marks)

**Goal:** a plan detailed enough that someone else could build it.

- **Decompose** the problem (structure chart / module breakdown).
- **Algorithms** for the key parts — in flowcharts/pseudocode (this is where your computational complexity shows).
- **Data structures & data** — variables, records, files/database schema, and *why* each choice.
- **UI design** — wireframes/screen designs and navigation.
- **Usability features** for the stakeholder.
- **A detailed test plan** — typical/boundary/erroneous data, mapped to success criteria.

**Top-band tip:** *justify* each design decision against the analysis. Designs should clearly connect to requirements.

---

## Stage 3 — Development & Testing (~25 marks — the biggest stage)

**Goal:** an iterative, well-evidenced build of a working solution.

- Develop in **clear iterations**; for each: what you built, **annotated code screenshots**, and **testing of that part** with evidence.
- Show **good programming practice**: meaningful names, modular subroutines, validation, error handling, comments.
- Evidence **testing throughout** (not just at the end) — show failures found and fixed.
- Demonstrate the **complexity** claimed in design (the actual algorithm/data structure working).
- Keep a development log so the **iterative process is visible** — examiners reward evidence of review and refinement between iterations.

**Top-band tip:** the narrative should read as *develop → test → review → refine*, repeatedly. Robustness (handles bad input without crashing) is heavily rewarded.

---

## Stage 4 — Evaluation (~20 marks)

**Goal:** honest, evidence-based judgement against your own success criteria.

- **Test against each success criterion** with evidence (screenshots/test table), including **boundary and erroneous** data.
- **Evaluate** how well each requirement was met — including ones only **partially** met (honesty scores).
- Get **stakeholder feedback** and use it.
- **Usability** evaluation.
- **Improvements / future development** — specific and justified.

**Top-band tip:** explicitly reference back to the **measurable** criteria from Analysis. A vague "it works well" stalls in the middle band; "criterion 4 was met — see Test 12; criterion 7 partially met because…" hits the top.

---

## Common ways students lose easy marks
- Vague, **unmeasurable success criteria** → weak evaluation later.
- **Generic stakeholder** ("any user") instead of a named one with real needs.
- Testing only at the **end** instead of throughout development.
- Code screenshots with **no annotation/explanation**.
- Project too **simple** (caps the achievable band) or too **big** (unfinished/no evaluation).
- Evaluation that doesn't **map back** to the analysis criteria.
- Poor **robustness** — crashes on boundary/erroneous input.

---

## A\* checklist for the project

- [ ] Named stakeholder, interviewed, needs documented
- [ ] Problem clearly justified as computationally solvable, with genuine algorithmic complexity
- [ ] Measurable, testable success criteria
- [ ] Decomposition + algorithms (pseudocode/flowcharts) + justified data structures + test plan
- [ ] Iterative development with annotated code and testing each iteration
- [ ] Robust validation/error handling; complexity actually implemented
- [ ] Evaluation against every success criterion with evidence, plus stakeholder feedback
- [ ] Justified improvements

Plan early, keep evidence as you go, and map everything back to your success criteria. Done well, this is the most reliable 20% on the whole A Level.
