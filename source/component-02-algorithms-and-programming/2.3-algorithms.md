## What you need to be able to do

This is one of the most heavily examined and **discriminating** topics on the paper. The specification requires you to be able to:

- [ ] **Analyse and design** algorithms for a given situation.
- [ ] Evaluate the **suitability of different algorithms** for a given task and data set, in terms of **execution time** and **space**.
- [ ] Understand the **measures and methods used to determine the efficiency** of algorithms.
- [ ] Use **Big O notation** to express time and space complexity: constant **O(1)**, linear **O(n)**, polynomial **O(n²)**, exponential **O(2ⁿ)** and logarithmic **O(log n)**.
- [ ] **Compare** the complexity of algorithms.
- [ ] Know the algorithms for the main **data structures** (stacks, queues, trees, linked lists), and **depth-first (post-order)** and **breadth-first** traversal of trees and graphs.
- [ ] Know how each **standard algorithm** works, **trace** it on data, and state its **Big O complexity**:
  - Bubble sort, insertion sort, merge sort, quick sort
  - Binary search, linear search
  - **Dijkstra's** shortest path algorithm
  - **A\*** algorithm

> 💡 **A\* extension — exam mindset:** OCR examiners reward candidates who can *justify* a choice of algorithm, not just name one. Whenever you state a Big O, be ready to explain *why* (e.g. "binary search halves the search space each pass, so the number of comparisons grows logarithmically with n").

### Big O notation

**Big O notation** describes how the **time** (number of operations) or **space** (memory) required by an algorithm **grows as the size of the input data (n) increases**. It expresses the **worst-case** asymptotic behaviour, ignoring constants and lower-order terms because we care about *growth*, not exact counts.

> We drop constants and lower-order terms: `3n² + 5n + 17` simplifies to **O(n²)** because for large `n` the `n²` term dominates everything else.

#### The five complexity classes you must know

| Class | Name | Behaviour as n grows | Typical example |
|-------|------|----------------------|-----------------|
| **O(1)** | Constant | Time does **not** change with n | Accessing `array[i]`; push/pop on a stack |
| **O(log n)** | Logarithmic | Grows very slowly; doubling n adds one step | Binary search |
| **O(n)** | Linear | Doubling n doubles the time | Linear search; single loop over a list |
| **O(n²)** | Polynomial (quadratic) | Doubling n quadruples the time | Bubble/insertion sort; nested loops |
| **O(2ⁿ)** | Exponential | Adding **one** item **doubles** the time | Naïve recursive subset/Fibonacci; brute-force travelling salesman |

> Note: O(n²) is the specific *polynomial* case named in the spec, but polynomial means any O(nᵏ) for constant k (e.g. O(n³)). Exponential O(2ⁿ) is far worse than any polynomial.

#### Graph description (how the curves look)

Imagine plotting **operations (y-axis)** against **input size n (x-axis)**:

```
operations
   |                                       O(2^n)   O(n^2)
   |                                      /        /
   |                                    /        /
   |                                  /       /
   |                                /      /        O(n)
   |                              /     /        /
   |                            /    /       /
   |                          /   /     /            ____ O(log n)
   |                        / /  /  / ____________----
   |__________________________________________________  O(1)
   +-------------------------------------------------> n
```

- **O(1)** is a flat horizontal line — never changes.
- **O(log n)** rises steeply at first then flattens — very efficient for large n.
- **O(n)** is a straight diagonal line.
- **O(n²)** is an upward-curving parabola.
- **O(2ⁿ)** rockets almost vertically — quickly becomes **intractable** (impossible to compute in reasonable time).

#### Comparison: number of operations for different n

| n | O(1) | O(log n) | O(n) | O(n²) | O(2ⁿ) |
|---|------|----------|------|-------|-------|
| 1 | 1 | 0 | 1 | 1 | 2 |
| 8 | 1 | 3 | 8 | 64 | 256 |
| 16 | 1 | 4 | 16 | 256 | 65,536 |
| 64 | 1 | 6 | 64 | 4,096 | ~1.8 × 10¹⁹ |

> 💡 **A\* extension — tractable vs intractable:** Problems solvable in **polynomial time or better** are considered *tractable*. Problems whose only known solutions are **exponential** (O(2ⁿ)) or worse are *intractable* for large n — even a fast computer cannot finish in a useful time. This links directly to spec 2.3.2 (Thinking ahead / complexity).

### Searching algorithms

#### Linear search

Checks each element in turn from the start until it finds the target (or reaches the end). Works on **unsorted** data.

```
FUNCTION linearSearch(list, target)
    FOR i = 0 TO LEN(list) - 1
        IF list[i] == target THEN
            RETURN i          // found at index i
        ENDIF
    NEXT i
    RETURN -1                 // not found
ENDFUNCTION
```

**Trace — search for 7 in `[3, 8, 2, 7, 5]`:**

| Step | i | list[i] | list[i] == 7? |
|------|---|---------|---------------|
| 1 | 0 | 3 | No |
| 2 | 1 | 8 | No |
| 3 | 2 | 2 | No |
| 4 | 3 | 7 | **Yes → return 3** |

- **Best case:** O(1) — target is the first element.
- **Average / Worst case:** **O(n)** — target is last or absent; must check every element.
- **Space:** O(1).

#### Binary search

Works only on a **sorted** list. Repeatedly examines the **middle** element and discards the half that cannot contain the target, **halving** the search space each pass.

```
FUNCTION binarySearch(list, target)
    low = 0
    high = LEN(list) - 1
    WHILE low <= high
        mid = (low + high) DIV 2
        IF list[mid] == target THEN
            RETURN mid
        ELSEIF list[mid] < target THEN
            low = mid + 1       // discard left half
        ELSE
            high = mid - 1      // discard right half
        ENDIF
    ENDWHILE
    RETURN -1                   // not found
ENDFUNCTION
```

**Trace — search for 23 in `[4, 8, 15, 16, 23, 42]` (indices 0–5):**

| Step | low | high | mid | list[mid] | Comparison | Action |
|------|-----|------|-----|-----------|------------|--------|
| 1 | 0 | 5 | 2 | 15 | 15 < 23 | low = 3 |
| 2 | 3 | 5 | 4 | 23 | 23 == 23 | **return 4** |

- **Best case:** O(1) — target is the middle element first time.
- **Average / Worst case:** **O(log n)** — search space halves each step.
- **Space:** O(1) iterative; O(log n) if implemented recursively (call stack).

> 💡 **A\* extension — when does linear beat binary?** Binary search needs a **sorted** list. If the data is unsorted and you only search once, sorting first (O(n log n)) then binary searching costs more than a single O(n) linear search. Binary search pays off when you **search the same sorted data many times**.

### Sorting algorithms

#### Bubble sort

Repeatedly steps through the list, **comparing adjacent pairs** and **swapping** them if they are in the wrong order. After each pass the largest unsorted element "bubbles" to the end. A `swapped` flag lets it stop early if a pass makes no swaps.

```
PROCEDURE bubbleSort(list)
    n = LEN(list)
    REPEAT
        swapped = False
        FOR i = 0 TO n - 2
            IF list[i] > list[i + 1] THEN
                temp = list[i]
                list[i] = list[i + 1]
                list[i + 1] = temp
                swapped = True
            ENDIF
        NEXT i
        n = n - 1          // last element now in place
    UNTIL swapped == False
ENDPROCEDURE
```

**Trace on `[5, 1, 4, 2]`:**

| Pass | Comparisons & swaps | List after pass |
|------|---------------------|-----------------|
| 1 | (5,1)→swap, (5,4)→swap, (5,2)→swap | `[1, 4, 2, 5]` |
| 2 | (1,4)→no, (4,2)→swap | `[1, 2, 4, 5]` |
| 3 | (1,2)→no, (2,4)→no (no swaps) | `[1, 2, 4, 5]` ✓ sorted |

#### Insertion sort

Builds a sorted section at the front. Takes the next element and **inserts** it into its correct position among the already-sorted elements by shifting larger ones right.

```
PROCEDURE insertionSort(list)
    FOR i = 1 TO LEN(list) - 1
        key = list[i]
        j = i - 1
        WHILE j >= 0 AND list[j] > key
            list[j + 1] = list[j]    // shift right
            j = j - 1
        ENDWHILE
        list[j + 1] = key            // insert key
    NEXT i
ENDPROCEDURE
```

**Trace on `[5, 1, 4, 2]`:**

| i | key | Before | Action | After |
|---|-----|--------|--------|-------|
| 1 | 1 | `[5,1,4,2]` | shift 5 right, insert 1 | `[1,5,4,2]` |
| 2 | 4 | `[1,5,4,2]` | shift 5 right, insert 4 | `[1,4,5,2]` |
| 3 | 2 | `[1,4,5,2]` | shift 5,4 right, insert 2 | `[1,2,4,5]` ✓ |

#### Merge sort

A **divide-and-conquer** algorithm. **Recursively splits** the list in half until each sublist has one element (trivially sorted), then **merges** sublists back together in sorted order.

```
FUNCTION mergeSort(list)
    IF LEN(list) <= 1 THEN
        RETURN list
    ENDIF
    mid = LEN(list) DIV 2
    left = mergeSort(list[0 .. mid-1])
    right = mergeSort(list[mid .. end])
    RETURN merge(left, right)
ENDFUNCTION

FUNCTION merge(left, right)
    result = []
    WHILE left and right both non-empty
        IF left[0] <= right[0] THEN
            append left[0] to result; remove from left
        ELSE
            append right[0] to result; remove from right
        ENDIF
    ENDWHILE
    append any remaining items
    RETURN result
ENDFUNCTION
```

**Trace on `[5, 1, 4, 2]`:**

```
Split:   [5, 1, 4, 2]
        /            \
     [5, 1]        [4, 2]
     /    \        /    \
   [5]   [1]     [4]   [2]
     \    /        \    /
Merge:[1, 5]      [2, 4]
        \            /
     Merge → [1, 2, 4, 5] ✓
```

Merging `[1,5]` and `[2,4]`: compare 1 vs 2 → take 1; 5 vs 2 → take 2; 5 vs 4 → take 4; take remaining 5 → `[1, 2, 4, 5]`.

#### Quick sort

Also **divide-and-conquer**. Chooses a **pivot**, **partitions** the list so smaller items go left and larger go right, then **recursively** quicksorts each partition.

```
FUNCTION quickSort(list)
    IF LEN(list) <= 1 THEN
        RETURN list
    ENDIF
    pivot = list[0]
    left  = [items in list[1..] that are < pivot]
    right = [items in list[1..] that are >= pivot]
    RETURN quickSort(left) + [pivot] + quickSort(right)
ENDFUNCTION
```

**Trace on `[5, 1, 4, 2]` (pivot = first element):**

```
quickSort([5,1,4,2])  pivot=5  → left=[1,4,2]  right=[]
  quickSort([1,4,2])  pivot=1  → left=[]   right=[4,2]
    quickSort([4,2])  pivot=4  → left=[2]  right=[]
      → [2] + [4] + [] = [2,4]
    → [] + [1] + [2,4] = [1,2,4]
  → [1,2,4] + [5] + [] = [1,2,4,5] ✓
```

#### Big O comparison table

| Algorithm | Best | Average | Worst | Space | Stable? | Notes |
|-----------|------|---------|-------|-------|---------|-------|
| **Bubble sort** | O(n) | O(n²) | O(n²) | O(1) | Yes | Best O(n) only with swap flag on nearly-sorted data; simple but slow |
| **Insertion sort** | O(n) | O(n²) | O(n²) | O(1) | Yes | Very efficient on small/nearly-sorted lists; in-place |
| **Merge sort** | O(n log n) | O(n log n) | O(n log n) | O(n) | Yes | Consistent; needs extra memory for merging |
| **Quick sort** | O(n log n) | O(n log n) | O(n²) | O(log n) | No (typically) | Worst case O(n²) on already-sorted data with poor pivot; usually fastest in practice |

> 💡 **A\* extension — why merge sort beats quicksort in the worst case:** Quick sort degrades to **O(n²)** when the pivot repeatedly splits the list unevenly (e.g. always picking the smallest/largest element, as on an already-sorted list). Merge sort **guarantees O(n log n)** every time because it always splits exactly in half. The trade-off: merge sort needs **O(n) extra space** for its temporary arrays, whereas quick sort sorts **in place** with only O(log n) recursion-stack space. This space-vs-time trade-off is exactly the kind of justification examiners reward.

### Data structure algorithms

#### Stack (LIFO — Last In, First Out)

Items are added and removed from the same end (the **top**). Implemented with an array and a `top` pointer.

```
PROCEDURE push(item)
    IF top == maxSize - 1 THEN
        RETURN "Stack overflow"
    ENDIF
    top = top + 1
    stack[top] = item
ENDPROCEDURE

FUNCTION pop()
    IF top == -1 THEN
        RETURN "Stack underflow"
    ENDIF
    item = stack[top]
    top = top - 1
    RETURN item
ENDFUNCTION

FUNCTION peek()         // look at top without removing
    RETURN stack[top]
ENDFUNCTION
```

- **push / pop / peek / isEmpty:** all **O(1)**.

#### Queue (FIFO — First In, First Out)

Items are added at the **rear (enQueue)** and removed from the **front (deQueue)**. A **circular queue** wraps pointers using `MOD` to reuse freed space.

```
PROCEDURE enQueue(item)         // linear queue
    IF rear == maxSize - 1 THEN
        RETURN "Queue full"
    ENDIF
    rear = rear + 1
    queue[rear] = item
ENDPROCEDURE

FUNCTION deQueue()
    IF front > rear THEN
        RETURN "Queue empty"
    ENDIF
    item = queue[front]
    front = front + 1
    RETURN item
ENDFUNCTION
```

- **enQueue / deQueue:** **O(1)**.

#### Linked list

A dynamic structure: each **node** holds **data** and a **pointer** to the next node. A `start` pointer marks the head; a null pointer marks the end. (A *free list* tracks unused nodes.)

```
FUNCTION searchList(start, target)        // O(n)
    current = start
    WHILE current != NULL
        IF current.data == target THEN
            RETURN current
        ENDIF
        current = current.pointer
    ENDWHILE
    RETURN NULL
ENDFUNCTION

PROCEDURE insertAfter(node, newNode)      // O(1) given the node
    newNode.pointer = node.pointer
    node.pointer = newNode
ENDPROCEDURE
```

- **Traversal / search:** O(n). **Insert/delete given the node:** O(1) (just re-point), versus O(n) shifting in an array.

#### Trees and tree traversals

A **binary tree** node has data, a **left** pointer and a **right** pointer. In a **binary search tree (BST)**, left subtree values < node < right subtree values. There are three **depth-first** traversals plus one **breadth-first**.

**Example tree:**

```
            50
          /    \
        30      70
       /  \    /  \
     20   40  60   80
```

**Pre-order (Node → Left → Right):** visit root first.
`50, 30, 20, 40, 70, 60, 80` — used to **copy** a tree / produce prefix expressions.

**In-order (Left → Node → Right):** visit left subtree first.
`20, 30, 40, 50, 60, 70, 80` — outputs a BST in **ascending sorted order**.

**Post-order (Left → Right → Node):** visit root last. *(This is the depth-first traversal the spec names explicitly.)*
`20, 40, 30, 60, 80, 70, 50` — used to **delete** a tree, evaluate **Reverse Polish (postfix)** expressions, and in **infix → postfix** conversion.

```
PROCEDURE postOrder(node)
    IF node != NULL THEN
        postOrder(node.left)
        postOrder(node.right)
        OUTPUT node.data
    ENDIF
ENDPROCEDURE
```

**Breadth-first traversal (BFS):** visit nodes **level by level**, left to right, using a **queue**. From the tree above: `50, 30, 70, 20, 40, 60, 80`.

```
PROCEDURE breadthFirst(root)
    enQueue(root)
    WHILE queue not empty
        node = deQueue()
        OUTPUT node.data
        IF node.left  != NULL THEN enQueue(node.left)  ENDIF
        IF node.right != NULL THEN enQueue(node.right) ENDIF
    ENDWHILE
ENDPROCEDURE
```

- **Depth-first** uses a **stack** (or recursion's call stack); **breadth-first** uses a **queue**.
- BST search/insert: **O(log n)** average (balanced), **O(n)** worst (degenerate/unbalanced tree). Traversal visits every node: **O(n)**.

> 💡 **A\* extension — DFS vs BFS for graphs:** On a graph, **depth-first search** dives as deep as possible before backtracking (good for maze/path existence, topological sorting); **breadth-first search** explores all neighbours first (good for finding the **shortest path in an unweighted graph**). Both are O(V + E) using an adjacency list. Remember to mark nodes as **visited** to avoid infinite loops on cyclic graphs.

### Path-finding: Dijkstra and A\*

#### Dijkstra's shortest path algorithm

Finds the shortest path from a **start node** to **all other nodes** in a **weighted graph** with non-negative edge weights. It keeps a tentative distance for every node, always **visits the unvisited node with the smallest tentative distance next**, and relaxes its neighbours.

**Algorithm outline:**

1. Set the start node's distance to 0, all others to ∞. Mark all unvisited.
2. From unvisited nodes, pick the one with the **smallest tentative distance** (current node).
3. For each unvisited neighbour, calculate `distance(current) + edge weight`. If this is **less** than the neighbour's recorded distance, **update** it (and record current as its predecessor).
4. Mark the current node **visited** (its distance is now final).
5. Repeat from step 2 until the destination is visited (or all nodes done).

**Worked example — find shortest path A → E:**

```
        4         3
   A --------B--------C
   |        /|        |
  1|     2/ |6        |2
   |    /   |         |
   D --------+--------E
        5         (D-E = 7)

Edges (undirected, weighted):
A-B = 4,  A-D = 1,  B-C = 3,  B-D = 2,  B-E = 6,  C-E = 2,  D-E = 7
```

**Table trace** (distances from A; "—" = settled/visited):

| Step | Current | A | B | C | D | E | Visited set |
|------|---------|---|---|---|---|---|-------------|
| Init | – | 0 | ∞ | ∞ | ∞ | ∞ | {} |
| 1 | **A** (0) | — | 4 | ∞ | **1** | ∞ | {A} |
| 2 | **D** (1) | — | min(4, 1+2)=**3** | ∞ | — | min(∞,1+7)=8 | {A,D} |
| 3 | **B** (3) | — | — | min(∞,3+3)=**6** | — | min(8, 3+6)=**9** | {A,D,B} |
| 4 | **C** (6) | — | — | — | — | min(9, 6+2)=**8** | {A,D,B,C} |
| 5 | **E** (8) | — | — | — | — | — | {A,D,B,C,E} |

**Shortest distance A → E = 8.** Tracing predecessors back: E ← C ← B ← D ← A, so the **path is A → D → B → C → E** (1 + 2 + 3 + 2 = 8).

- **Complexity:** O(n²) with a simple array of distances; O((V + E) log V) with a priority queue (binary heap).

#### A\* algorithm

A\* is an **extension/optimisation of Dijkstra** for finding the shortest path between a **single start and a single goal**. It is **informed**: as well as the actual cost so far, it uses a **heuristic** that *estimates* the remaining cost to the goal, so it explores towards the target rather than in all directions.

For each node it computes:

```
f(n) = g(n) + h(n)
```

- **g(n)** = the actual cost from the start node to n (as in Dijkstra).
- **h(n)** = the **heuristic** — an *estimate* of the cost from n to the goal (e.g. straight-line / Manhattan distance).
- **f(n)** = estimated total cost of the cheapest path through n. A\* always expands the node with the **lowest f(n)** next.

The heuristic must be **admissible** (never *overestimates* the true remaining cost) for A\* to guarantee the optimal path.

#### Dijkstra vs A\* comparison

| | Dijkstra | A\* |
|---|----------|-----|
| Cost function | g(n) only | f(n) = g(n) + h(n) |
| Uses a heuristic? | No (uninformed) | **Yes** (informed) |
| Searches | Outward in all directions | Directed toward the goal |
| Typical use | Shortest path to **all** nodes | Shortest path to **one specific** goal |
| Efficiency | Explores more nodes | Usually explores **fewer** nodes → faster |
| Optimality | Always optimal (non-negative weights) | Optimal **if** heuristic is admissible |

> 💡 **A\* extension — the heuristic is everything:** If `h(n) = 0` for all nodes, A\* behaves **identically to Dijkstra**. A good heuristic (close to the true remaining distance) makes A\* dramatically faster by reducing the number of nodes expanded — this is why A\* is the workhorse of game pathfinding and sat-nav routing. But if the heuristic *overestimates*, A\* may return a non-optimal path. There is a **time–accuracy trade-off** in choosing the heuristic.

## Key terms (exam definitions)

| Term | Definition |
|------|------------|
| **Algorithm** | A precise, finite sequence of steps to solve a problem or perform a task. |
| **Big O notation** | A notation expressing how an algorithm's time or space requirement grows with the size of the input (n), in the worst case. |
| **Time complexity** | A measure of the number of operations an algorithm performs as a function of input size. |
| **Space complexity** | A measure of the amount of memory an algorithm needs as a function of input size. |
| **Constant time — O(1)** | Execution time independent of input size. |
| **Linear time — O(n)** | Execution time grows in direct proportion to input size. |
| **Polynomial time — O(n²)** | Execution time grows with the square (or higher power) of the input size. |
| **Exponential time — O(2ⁿ)** | Execution time doubles with each additional input element; intractable for large n. |
| **Logarithmic time — O(log n)** | Execution time grows by a constant amount each time n doubles. |
| **Tractable** | A problem that has a solution in polynomial time or better. |
| **Intractable** | A problem with no known polynomial-time solution (only exponential or worse). |
| **Divide and conquer** | Breaking a problem into smaller subproblems, solving them, and combining results (merge/quick sort, binary search). |
| **Pivot** | The element chosen in quick sort to partition the list. |
| **Stable sort** | A sort that preserves the relative order of equal elements. |
| **Heuristic** | A "rule of thumb" estimate; in A\*, an estimate h(n) of the remaining cost to the goal. |
| **Admissible heuristic** | A heuristic that never overestimates the true remaining cost, guaranteeing A\* optimality. |
| **Traversal** | Visiting every node of a structure in a defined order (pre/in/post-order, BFS). |

## Common exam mistakes & examiner tips

- ❌ **Confusing the traversal orders.** Remember by the position of the **Node**: **Pre**-order = Node first, **In**-order = Node in the middle (gives sorted BST output), **Post**-order = Node last. Examiners frequently ask for in-order to demonstrate sorted output.
- ❌ **Stating Big O without justification.** A 4-mark question asks you to *explain why* — e.g. "O(n²) because of the nested loop, where each of n passes makes up to n comparisons."
- ❌ **Giving binary search's complexity as O(n).** It is **O(log n)** — and only works on **sorted** data. State the precondition.
- ❌ **Forgetting the swap flag** when claiming bubble sort's best case is O(n). Without it, bubble sort is O(n²) even on a sorted list.
- ❌ **Saying quick sort is always faster than merge sort.** Quick sort's **worst case is O(n²)**; merge sort guarantees O(n log n) but uses more memory.
- ❌ **Mixing up the data structures used by DFS and BFS.** Depth-first uses a **stack**; breadth-first uses a **queue**.
- ❌ **In Dijkstra, updating already-visited (settled) nodes.** Once a node is marked visited its distance is final.
- ❌ **Describing A\* without the heuristic.** The key distinction from Dijkstra is `f(n) = g(n) + h(n)`; always mention the heuristic and admissibility.
- ✅ **Tip:** When tracing an algorithm, **show a table** of variable values per pass/step — examiners award marks for the working, not just the final answer.
- ✅ **Tip:** Learn to **complete-the-algorithm** questions: read the surrounding pseudocode carefully and match its conventions (e.g. `DIV`, `MOD`, indexing from 0).

**Q4 (AO1/AO2, 6 marks).** Compare **merge sort** and **quick sort** in terms of time complexity and space complexity. Justify which you would choose to sort a very large list where worst-case performance must be guaranteed.

<details><summary>Mark scheme</summary>

- Merge sort time complexity is **O(n log n)** in all cases (best/average/worst) (1).
- Quick sort is O(n log n) average but **O(n²) worst case** (e.g. poor pivot / already-sorted data) (1).
- Quick sort sorts **in place** — space O(log n) (recursion) (1); merge sort needs **O(n) extra space** for merging (1).
- Choose **merge sort** because it **guarantees O(n log n)** even in the worst case (1).
- Accept trade-off discussion: this costs extra memory, acceptable if worst-case time is the priority (1).
</details>

**Q7 (AO1, 4 marks).** Explain how the **A\*** algorithm differs from Dijkstra's algorithm, including the role of the **heuristic**.

<details><summary>Mark scheme</summary>

- A\* uses a cost function **f(n) = g(n) + h(n)** (1) where g is the actual cost so far and **h is a heuristic estimate of the remaining cost to the goal** (1).
- A\* is **informed/directed toward the goal**, so it usually explores fewer nodes / is faster than Dijkstra which searches outward in all directions (1).
- A\* remains optimal provided the heuristic is **admissible (does not overestimate)** / if h(n)=0 it reduces to Dijkstra (1).
</details>

**Q9 (AO3, 6 marks).** A program must repeatedly search a dataset of 1,000,000 records. Discuss whether **linear search** or **binary search** is more suitable, including any preconditions and the impact of sorting cost.

<details><summary>Mark scheme</summary>

- Linear search is **O(n)** — up to 1,000,000 comparisons; works on **unsorted** data (1).
- Binary search is **O(log n)** — about 20 comparisons; **requires sorted data** (1).
- Sorting once costs O(n log n) (1).
- If the data is searched **many times**, sorting once then binary searching repeatedly is **far more efficient** overall (1).
- If searched **only once** on unsorted data, a single linear search (O(n)) may be cheaper than sorting first (1).
- Reasoned conclusion appropriate to "repeatedly" → choose **binary search** (sort once, search many) (1).
</details>
