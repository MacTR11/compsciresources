# 2.2.1 Programming techniques

### Programming constructs

All programs, no matter how complex, are built from three fundamental constructs.

**Sequence** — instructions are carried out one after another, in the order they are written. The order matters: swapping two lines can change the result.

```
total = price + tax
total = total * quantity
print(total)
```

**Selection (branching)** — the program chooses between different paths depending on a condition. Implemented using `if`/`elseif`/`else` or `switch`/`case`.

```
if score >= 70 then
    print("Distinction")
elseif score >= 50 then
    print("Pass")
else
    print("Fail")
endif
```

```
switch grade:
    case "A":
        print("Excellent")
    case "B":
        print("Good")
    default:
        print("Keep trying")
endswitch
```

**Iteration (looping)** — a block of code is repeated. There are two kinds:

- **Count-controlled iteration** — repeats a *fixed, known* number of times. Use a `for` loop.

```
for i = 1 to 5
    print("Hello " + i)
next i
```

- **Condition-controlled iteration** — repeats *until a condition changes*; the number of repeats is not known in advance. Use `while` (condition tested at the **start** — may run zero times) or `do...until` (condition tested at the **end** — always runs at least once).

```
while password != "letmein"
    password = input("Enter password: ")
endwhile
```

```
do
    answer = input("Continue? (y/n): ")
until answer == "n"
```

> 💡 **A\* extension:** A `while` loop is **pre-conditioned** (tests before the body) so its body may execute *zero* times; a `do...until` loop is **post-conditioned** (tests after the body) so it always executes *at least once*. Knowing which to choose is a common AO2 design decision — e.g. input validation usually wants `do...until` because you must ask at least once.

### Recursion

**Recursion** is when a subroutine (function or procedure) **calls itself** in order to solve a problem. A correctly written recursive routine must have:

1. A **base case** — a condition that stops the recursion (no further recursive call). Without it, recursion never ends and causes a **stack overflow**.
2. A **general/recursive case** — the routine calls itself with a value closer to the base case.

```
function factorial(n)
    if n == 0 then          // base case
        return 1
    else
        return n * factorial(n - 1)   // recursive case
    endif
endfunction
```

**How the stack is used:** every time a subroutine is called, a **stack frame** is pushed onto the **call stack**. The frame stores the parameters, local variables and the **return address** (where to continue after the call returns). Recursive calls keep pushing frames until the base case is reached; then frames are **popped** one at a time as each call returns, in reverse order (Last In, First Out).

**When to use recursion:** when a problem is naturally **self-similar** — it can be defined in terms of smaller versions of itself. Examples: traversing trees, the factorial/Fibonacci definitions, quicksort, merge sort, navigating folder structures, the Towers of Hanoi.

**Recursion vs iteration:**

| Aspect | Recursion | Iteration |
|---|---|---|
| Memory | Uses more (each call adds a stack frame) | Uses less (one set of variables) |
| Speed | Generally slower (call overhead) | Generally faster |
| Code length | Often shorter/elegant for self-similar problems | Can be longer for tree-like problems |
| Risk | **Stack overflow** if base case wrong/too deep | Infinite loop if condition never met |
| Suitability | Tree/recursive structures | Linear, repeated tasks |

Any recursive solution can be rewritten iteratively (often using an explicit stack), and vice versa.

> 💡 **A\* extension:** Recursion that does its work *after* the recursive call returns cannot be optimised away. Some languages support **tail recursion**, where the recursive call is the last operation, allowing the compiler to reuse one stack frame. OCR rarely names this, but understanding *why* recursion is memory-hungry (one frame per active call) earns marks.

### Global vs local variables and scope

The **scope** of a variable is **the region of the program in which that variable can be accessed (is "visible")**.

- A **local variable** is declared inside a subroutine. It exists only while that subroutine runs and can only be accessed within it. Two subroutines can each have a local variable with the same name without clashing.
- A **global variable** is declared in the main program, outside any subroutine, and can be accessed (and changed) anywhere in the program.

```
global highScore = 0          // global – visible everywhere

procedure updateScore(points)
    bonus = points * 2        // local – only exists in this procedure
    highScore = highScore + bonus
endprocedure
```

**Why prefer local variables?**
- They avoid accidental changes from other parts of the program (fewer side effects).
- They free up memory when the subroutine ends.
- They make code more **modular**, reusable and easier to debug.

**Why globals are sometimes needed:** for values genuinely shared across the whole program (e.g. a configuration setting or a running total used everywhere). Overusing globals makes programs hard to maintain.

> 💡 **A\* extension:** If a local variable has the same name as a global, the **local takes precedence** within its subroutine (it "shadows" the global). To deliberately change a global inside a subroutine, many languages require an explicit declaration (e.g. Python's `global` keyword); in OCR pseudocode you simply assume the global is visible.

---

### Modularity, functions and procedures

**Modularity** is the design technique of breaking a large program into smaller, self-contained **modules (subroutines)**, each performing one specific task. Benefits:

- Easier to **write, test and debug** (each module tested independently).
- Modules can be **reused** in other programs.
- A **team** can work on different modules simultaneously.
- The program is easier to **read and maintain**.

A **subroutine** is a named block of code that performs a task and can be called by name. There are two types:

- A **procedure** performs a task but does **not** return a value.
- A **function** performs a task and **returns a single value** to the point where it was called.

```
procedure greet(name)             // procedure – no return value
    print("Hello " + name)
endprocedure

function square(x)                 // function – returns a value
    return x * x
endfunction
```

A function call can be used inside an expression (`y = square(4) + 1`); a procedure call is a statement on its own (`greet("Ada")`).

> 💡 **A\* extension:** In strict terms a function should be **side-effect free** — it computes a result purely from its parameters and changes nothing else. This makes programs predictable and easier to test. Procedures, by contrast, exist *for* their side effects (printing, updating a file, changing a global).

---

### Parameters and parameter passing

A **parameter** is a variable listed in a subroutine's definition. An **argument** is the actual value supplied when the subroutine is called. Parameters let one subroutine work with different data each time.

There are two ways data is passed:

- **By value** — a **copy** of the argument's value is passed to the subroutine. Changes made inside the subroutine do **not** affect the original variable in the calling code.
- **By reference** — the **memory address (a reference)** of the original variable is passed. The subroutine works on the original, so changes made inside **do** affect the original variable.

```
procedure addOneByValue(byVal n)
    n = n + 1
    print(n)            // prints changed copy
endprocedure

procedure addOneByRef(byRef n)
    n = n + 1           // changes the ORIGINAL
endprocedure
```

By value is safer (no accidental side effects); by reference is efficient for large data (no copy made) and lets a subroutine return changes through its parameters.

> 💡 **A\* extension:** By reference avoids copying large structures (e.g. a 10,000-element array), saving both time and memory — but at the cost of safety, because the caller's data can be altered unexpectedly. Many languages pass primitive types by value and objects/arrays by reference by default.

### Use of an IDE

An **Integrated Development Environment (IDE)** is a single application providing the tools a programmer needs to write, test and debug code. Key features OCR expects you to know:

- **Editor with syntax highlighting** — colours keywords/strings to aid reading.
- **Code completion / auto-complete (intellisense)** — suggests names, speeds up typing, reduces typos.
- **Auto-indentation and pretty-printing** — formats code consistently.
- **Error diagnostics / error highlighting** — flags syntax errors as you type.
- **Debugging tools** — **breakpoints** (pause execution at a chosen line), **stepping** (run one line at a time), **variable watch / watch window** (inspect values while running).
- **Run-time environment / built-in interpreter or compiler** — run code without leaving the IDE.
- **Translator (compiler/interpreter)** integration.
- **Stepping through code** and a **call stack viewer**.

These features reduce development time and make finding and fixing errors faster.

### Object-oriented programming (OOP)

OOP models a program as a collection of **objects** that hold data and the operations that act on that data.

- A **class** is a **template/blueprint** that defines the **attributes** (data) and **methods** (behaviours/subroutines) that its objects will have.
- An **object** is a specific **instance** of a class, created at run time.
- An **attribute (field/property)** is a variable belonging to an object — its data.
- A **method** is a subroutine belonging to a class — its behaviour.
- A **constructor** is a special method that runs when an object is created, used to set up (initialise) its attributes. In OCR ERL it is often written as `new`.

```
class Animal
    private name
    private legs

    public procedure new(givenName, givenLegs)   // constructor
        name = givenName
        legs = givenLegs
    endprocedure

    public function getName()       // getter
        return name
    endfunction

    public procedure setName(newName)   // setter
        name = newName
    endprocedure

    public procedure speak()
        print("Some generic sound")
    endprocedure
endclass

myPet = new Animal("Rex", 4)        // creating an object (instantiation)
print(myPet.getName())
```

**Encapsulation** — bundling attributes and the methods that use them inside a class, and keeping attributes **private** so they can only be accessed/changed through **public methods**. This protects data integrity (data hiding).

**Getters and setters** — public methods that read (**getter/accessor**) and write (**setter/mutator**) private attributes in a controlled way. A setter can include **validation** before changing the value.

**Inheritance** — a class (the **subclass/child**) can **inherit** the attributes and methods of another class (the **superclass/parent**), then add or change its own. This promotes **reuse** and models "is-a" relationships. In OCR ERL: `inherits`.

```
class Dog inherits Animal
    public procedure new(givenName)
        super.new(givenName, 4)     // call parent constructor
    endprocedure

    public procedure speak()        // overriding the parent method
        print("Woof!")
    endprocedure
endclass

myDog = new Dog("Bella")
myDog.speak()        // outputs "Woof!"
```

**Polymorphism** — literally "many forms": the same method call behaves differently depending on the object's class. Above, `speak()` outputs a generic sound for an `Animal` but "Woof!" for a `Dog`. **Overriding** a method in a subclass is the most common form of polymorphism at A Level.

**Benefits of OOP:** reusable code (inheritance), data security (encapsulation), easier maintenance, and code that mirrors real-world objects, making large systems easier to design and divide among a team.

> 💡 **A\* extension:** Distinguish **overriding** (a subclass replaces an inherited method with the same name and parameters) from **overloading** (multiple methods share a name but have different parameter lists). Polymorphism via overriding lets you store mixed objects in one collection and call `speak()` on each, letting each object respond in its own way — powerful for extensible designs.

# 2.2.2 Computational methods

### Features that make a problem solvable by computational methods

A problem is suitable for a **computational solution** when it has features a computer can exploit. Look for:

- It can be **broken down (decomposed)** into smaller sub-problems.
- It involves **inputs, processes and outputs** that can be clearly defined.
- It is **repetitive** or follows rules/patterns that can be expressed as an algorithm.
- Useful **abstractions** can be made (irrelevant detail removed).
- The data and relationships involved can be **structured/modelled** (e.g. variables, arrays, objects).
- A **clear, finite sequence of steps** to reach a solution exists.

Not all problems are computable (e.g. the Halting Problem), and some are computable but **intractable** (would take impractically long). Recognising what *can* realistically be solved is part of **thinking computationally**.

---

### Problem recognition

**Problem recognition** is the first stage: clearly identifying and understanding **what the problem actually is** before attempting to solve it. It means defining the requirements — the inputs available, the outputs needed, the constraints, and what a successful solution looks like. Misunderstanding the problem leads to solving the wrong thing.

---

### Problem decomposition

**Decomposition** is breaking a complex problem down into **smaller, more manageable sub-problems** that can each be solved (and tested) individually. Each sub-problem may be broken down further. This makes a daunting task achievable, supports modular code and lets a team divide work.

*Example:* building a game — decompose into rendering graphics, handling user input, physics, scoring and AI; then break "scoring" into "detect collision", "update score", "display score".

---

### Divide and conquer

**Divide and conquer** is a problem-solving approach that repeatedly **divides** a problem in half (or into parts), solves each part, then **combines** the results. It often reduces a problem of size *n* so that each step only handles half the data, giving very efficient algorithms.

*Examples:* **binary search** (halve the search space each comparison), **merge sort** and **quicksort** (split the list, sort the parts, recombine). Closely linked to recursion.

> 💡 **A\* extension:** Decomposition and divide and conquer are related but distinct. **Decomposition** breaks a problem into *different* sub-tasks (input, process, output). **Divide and conquer** breaks a problem into *smaller instances of the same problem* and combines their solutions — which is why it pairs so naturally with recursion and gives logarithmic-style efficiency gains.

---

### Abstraction

**Abstraction** is the process of **removing or hiding unnecessary detail** so you can focus only on the information needed to solve the problem.

- **Representational abstraction** — removing details that are not needed (e.g. a Tube map shows the order of stations, not real distances).
- **Abstraction by generalisation** — grouping things by shared characteristics (e.g. treating all "vehicles" the same way regardless of make).

Abstraction makes complex problems manageable and models simpler. A program is itself an abstraction of a real-world process.

---

### Methods to solve problems

#### Backtracking

**Backtracking** is a method that builds a solution step by step and, whenever a step leads to a **dead end** (a partial solution that cannot be completed), it **goes back** to the previous decision point and tries a different option. It systematically explores possibilities, abandoning paths that fail.

*When used:* solving mazes, the **N-Queens** problem, **Sudoku** solvers, and finding paths in graphs. Often implemented recursively (the call stack records the choices to undo).

#### Data mining

**Data mining** is the process of **searching through large data sets (often "big data") to discover patterns, correlations and trends** that were not previously obvious, in order to gain useful information.

*When used:* a supermarket analysing loyalty-card transactions to find that customers who buy product A often buy product B, informing product placement and offers; banks detecting fraud patterns.

#### Heuristics

A **heuristic** is a **"rule of thumb"** — a practical, non-guaranteed approach that finds a **good enough** solution quickly when finding the perfect solution would take too long (or is impossible). It trades accuracy for speed.

*When used:* satnav route-finding (A\* search uses a distance heuristic), antivirus software flagging suspicious behaviour, the **Travelling Salesman Problem** where an exact answer is intractable for large inputs.

#### Performance modelling

**Performance modelling** is **using mathematical/computational models to predict and test how a system will perform** under different conditions, instead of (or before) building and stressing the real system — which may be expensive, dangerous or impossible to test directly.

*When used:* modelling how a new server will cope with peak traffic, testing aircraft systems, or predicting how an algorithm scales before deployment.

#### Pipelining

**Pipelining** is splitting a process into a series of **stages** so that, like a factory production line, different stages can work on **different items at the same time** — the output of one stage feeds the next. While stage 2 processes item 1, stage 1 can start item 2, increasing throughput.

*When used:* the **fetch–decode–execute** cycle in a CPU (instruction pipelining), and processing pipelines where data passes through transformation stages.

#### Visualisation

**Visualisation** is **representing data or a problem graphically** (charts, graphs, maps, diagrams, heat maps) to make patterns, trends and relationships easier for humans to see and understand, aiding analysis and decision-making.

*When used:* plotting sales as a graph to spot seasonal trends, a heat map of website clicks, or visualising a network/graph to find bottlenecks.

> 💡 **A\* extension:** These methods are not mutually exclusive — real solutions combine them. A satnav **decomposes** the journey, **abstracts** roads into a weighted graph, uses a **heuristic** (A\*) to search efficiently, may **backtrack** when a road is closed, and **visualises** the route on a map. In the exam, justify *why* a chosen method suits the problem (e.g. "a heuristic because an exact route is intractable for a large road network").

## Worked examples

### Example 1 — Recursive factorial traced with the call stack

```
function factorial(n)
    if n == 0 then
        return 1
    else
        return n * factorial(n - 1)
    endif
endfunction
```

Calling `factorial(3)`. Frames are **pushed** until the base case, then **popped** returning values:

```
PUSH  factorial(3)  -> needs 3 * factorial(2)
PUSH  factorial(2)  -> needs 2 * factorial(1)
PUSH  factorial(1)  -> needs 1 * factorial(0)
PUSH  factorial(0)  -> base case, returns 1     <-- stack at deepest point
```

Unwinding (popping):

| Call returns | Calculation | Value returned |
|---|---|---|
| factorial(0) | base case | 1 |
| factorial(1) | 1 * 1 | 1 |
| factorial(2) | 2 * 1 | 2 |
| factorial(3) | 3 * 2 | **6** |

Final answer: `factorial(3) = 6`. Note four frames existed on the stack simultaneously — this is why deep recursion risks **stack overflow**.

### Example 2 — By value vs by reference

```
procedure changeByValue(byVal x)
    x = 99
endprocedure

procedure changeByReference(byRef x)
    x = 99
endprocedure

num = 5
changeByValue(num)
print(num)            // outputs 5  (original unchanged – worked on a copy)

num = 5
changeByReference(num)
print(num)            // outputs 99 (original changed – worked on the variable itself)
```

**By value** passes a copy, so `num` is untouched. **By reference** passes the variable's address, so the procedure alters the original.

### Example 3 — Class definition with inheritance (and polymorphism)

```
class Shape
    private name

    public procedure new(givenName)
        name = givenName
    endprocedure

    public function area()
        return 0
    endfunction

    public function getName()
        return name
    endfunction
endclass

class Rectangle inherits Shape
    private width
    private height

    public procedure new(w, h)
        super.new("Rectangle")
        width = w
        height = h
    endprocedure

    public function area()              // overrides Shape.area – polymorphism
        return width * height
    endfunction
endclass

myShape = new Rectangle(4, 3)
print(myShape.getName())                // "Rectangle"  (inherited method)
print(myShape.area())                   // 12           (overridden method)
```

`Rectangle` **inherits** `getName()` and reuses `Shape`'s constructor via `super`, **encapsulates** `width`/`height` as private, and **overrides** `area()` to give correct behaviour (**polymorphism**).

### Example 4 — A computational method: backtracking (maze)

A backtracking maze solver: try a direction; if it reaches the exit, done; if it hits a wall or a visited cell, **backtrack** and try the next direction.

```
function solve(maze, row, col)
    if isExit(row, col) then
        return True                         // success
    endif
    if not isValidMove(maze, row, col) then
        return False                        // dead end – backtrack
    endif

    maze[row][col] = "visited"

    if solve(maze, row - 1, col) then return True endif   // up
    if solve(maze, row + 1, col) then return True endif   // down
    if solve(maze, row, col - 1) then return True endif   // left
    if solve(maze, row, col + 1) then return True endif   // right

    maze[row][col] = "unvisited"            // undo (backtrack)
    return False
endfunction
```

When all four directions fail, the cell is unmarked and `False` is returned, causing the *previous* call to try its next option — the essence of **backtracking**.

## Common exam mistakes & examiner tips

- **Function vs procedure:** state the *defining* difference clearly — a function **returns a value**, a procedure does not. Don't say "a function is more complex".
- **Recursion answers:** always mention a **base case** AND that it **calls itself**. For "why recursion can fail" name **stack overflow** caused by a missing/unreached base case.
- **Stack tracing:** show frames being **pushed then popped** in the correct **LIFO** order; many lose marks by computing the answer without showing the stack.
- **By value/by reference:** examiners want the consequence — *by value the original is unchanged; by reference it can be changed* — not just "a copy is made".
- **Scope:** don't confuse a global variable with a constant. Scope is about *visibility*, not whether the value can change.
- **OOP terms:** keep **encapsulation** (private data + public methods) separate from **inheritance** (reusing a parent) and **polymorphism** (same call, different behaviour). Mixing these up is the most common OOP error.
- **Computational methods:** for backtracking/heuristics etc., give a **named example** and **say why** the method suits that problem — bare definitions cap your marks.
- **Heuristic:** stress it gives a *good enough*, **not guaranteed optimal**, solution. That trade-off is the marking point.
- **Pipelining:** explain overlap (different stages work on different items *simultaneously*), not just "stages".
- **Always use OCR pseudocode** for code-writing questions: `if...endif`, `for...next`, `while...endwhile`, `procedure...endprocedure`, `function...return...endfunction`.

<details><summary>Mark scheme</summary>

- Recursion is a subroutine that calls itself (1).
- It must have a base case that stops the recursion (1) / and a recursive case that moves towards it (1).

**Q3 (AO2, 4 marks).** The function below is called as `mystery(4)`. Trace the calls using the call stack and state the value returned.

```
function mystery(n)
    if n <= 1 then
        return 1
    else
        return n + mystery(n - 1)
    endif
endfunction
```

<details><summary>Mark scheme</summary>

Calls pushed: mystery(4) → mystery(3) → mystery(2) → mystery(1) (base case) (1 for correct order of pushes).
Unwinding: mystery(1)=1; mystery(2)=2+1=3; mystery(3)=3+3=6; mystery(4)=4+6=10 (1 for correct unwinding).
- Correct identification of base case returning 1 (1).
- **Final value returned = 10** (1).
</details>

**Q4 (AO2, 6 marks).** Define a class `BankAccount` in OCR pseudocode. It should have private attributes `balance` and `owner`, a constructor, a method to deposit money, and a getter for the balance.

<details><summary>Mark scheme</summary>

Award up to 6, e.g.:
- `class BankAccount` with two **private** attributes (1).
- Constructor `new` taking parameters and assigning attributes (1–2).
- `deposit` method that adds an amount to `balance` (1).
- A getter `getBalance` that **returns** balance (1).
- Correct OCR syntax / encapsulation (private attributes accessed via methods) (1).

```
class BankAccount
    private balance
    private owner
    public procedure new(o, startBalance)
        owner = o
        balance = startBalance
    endprocedure
    public procedure deposit(amount)
        balance = balance + amount
    endprocedure
    public function getBalance()
        return balance
    endfunction
endclass
```
</details>

**Q5 (AO1, 4 marks).** Explain the difference between passing a parameter **by value** and **by reference**, giving one advantage of each.

<details><summary>Mark scheme</summary>

- By value: a copy of the data is passed (1); changes inside the subroutine do not affect the original (1) — advantage: safer / avoids unintended side effects.
- By reference: the memory address of the variable is passed (1); changes inside affect the original (1) — advantage: efficient for large data / no copy made / can return changes.
*(Max 4 across the two)*
</details>

**Q8 (AO3, 8 marks).** Discuss how the techniques of **decomposition, abstraction and divide and conquer** could be used to design a program that finds the shortest route between two stations on a railway network.

<details><summary>Mark scheme (levelled)</summary>

**Level 3 (6–8):** A coherent discussion correctly applying all three techniques to the railway scenario, with clear justification and good terminology.
**Level 2 (3–5):** Two techniques applied, or all three with some inaccuracy/limited links to the scenario.
**Level 1 (1–2):** Basic definitions with weak links to the problem.

Indicative content:
- **Decomposition:** split into sub-tasks — read the network data, build the model, search for a route, output/visualise it.
- **Abstraction:** represent stations as nodes and lines as weighted edges (a graph); ignore irrelevant detail such as platform layout or train colour; distances/times as edge weights.
- **Divide and conquer / search:** apply an algorithm (e.g. Dijkstra/binary-search-style narrowing) that breaks the search into smaller parts; could combine sub-route results.
- May reference heuristics (A\*) or visualisation of the final route for credit.
</details>

**Q9 (AO1, 3 marks).** Explain the term **encapsulation** and give one reason it is used in object-oriented programming.

<details><summary>Mark scheme</summary>

- Encapsulation is bundling attributes (data) and the methods that act on them inside a class (1).
- Attributes are kept private and accessed only through public methods (1).
- Reason: protects data integrity / data hiding / prevents invalid or accidental changes / validation can be applied in setters (1).
</details>
