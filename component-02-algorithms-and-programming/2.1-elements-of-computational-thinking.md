# 2.1 Elements of Computational Thinking

*OCR A Level Computer Science (H446) — Component 02: Algorithms and Programming — Spec reference 2.1.1–2.1.5*

> Computational thinking is the set of mental skills used to analyse a problem and design a solution that a computer can carry out. This section is **conceptual** and is assessed almost entirely by **application**: the exam gives you an unseen scenario (a game, a booking system, a robot, a website) and asks you to *apply* abstraction, thinking ahead, procedural, logical and concurrent thinking to it. Memorising definitions is necessary but **not sufficient** — you must be able to transfer the skill to a situation you have never seen.

---

## What you need to be able to do

- [ ] **2.1.1 Thinking abstractly** — explain the nature and need for abstraction; describe the differences between abstraction and reality; devise an abstract model for a variety of situations; distinguish *representational* abstraction from *generalisation/categorisation* abstraction.
- [ ] **2.1.2 Thinking ahead** — identify inputs and outputs for a given situation; determine the preconditions for devising a solution; explain the nature and benefits of caching; explain the need for and benefits of reusable program components.
- [ ] **2.1.3 Thinking procedurally** — identify the components of a problem; identify the components of a solution; identify the order in which steps must be performed; identify sub-procedures necessary to solve a problem (decomposition).
- [ ] **2.1.4 Thinking logically** — identify the points in a problem where a decision must be taken; determine how the flow of control is determined by those decisions; describe the conditions that lead to a particular outcome at each decision point.
- [ ] **2.1.5 Thinking concurrently** — determine the parts of a problem that can be tackled at the same time; explain the benefits and the trade-offs that come with concurrent processing.

## Revision notes by subsection

### 2.1.1 Thinking abstractly

**Abstraction** is the process of **removing or hiding unnecessary detail** so that only the information *relevant to the problem* remains. It lets us manage complexity and concentrate on what matters.

**The need for abstraction:** real-world problems contain far too much detail to model directly. By stripping away irrelevant facts we make a problem *solvable*, *understandable* and *reusable*. Abstraction also allows different people to work at different levels (e.g. a programmer uses a library without knowing its internal code).

**Differences between abstraction and reality:** an abstraction is a **simplified, idealised model**, not the real thing. It deliberately *omits* detail, so it will never behave exactly like reality. A model that captured every detail would be as complex as the original and useless for problem-solving.

> **Worked scenario — the London Underground map.** The Tube map is a famous abstraction. It *keeps* the order of stations, the line each station is on, and where lines interchange. It *removes* the real geography: actual distances, the curves of the tunnels, surface streets. It differs from reality because two stations shown close together may be far apart on the ground — but for the problem "which trains do I take to get from A to B?" that omitted detail is irrelevant.

There are **two types of abstraction** named in the spec:

- **Representational abstraction** — produced by *removing unnecessary details* from a particular thing to leave a simpler representation of that *specific* problem. *Example:* representing a delivery road network as a graph of nodes (junctions) and weighted edges (road lengths), ignoring traffic lights, surfaces and scenery.
- **Generalisation / categorisation abstraction** — grouping things together by their **shared characteristics** to make a more general, reusable model. *Example:* recognising that a "savings account" and a "current account" share attributes (balance, owner, account number) and methods (deposit, withdraw), so you create a general `Account` category that both inherit from.

> 💡 **A\* extension.** Layered abstraction is what makes large systems tractable: a programmer uses the TCP/IP model, an OS API, or a class library at one abstraction level without knowing the level beneath. The key examiner discriminator at the top band is *justifying which details are safe to discard* for a given problem — a good answer states the detail removed **and** why it is irrelevant to *this* problem, not just that "detail is removed". Be ready to argue that an over-abstracted model loses fidelity (a flight simulator that ignores wind is useless for training pilots in crosswinds).

### 2.1.2 Thinking ahead

Thinking ahead means planning a solution *before* coding by anticipating what it will need.

- **Identifying inputs and outputs.** Before designing, list every piece of data that must go *into* the system (**inputs**) and every result it must produce (**outputs**). Getting these wrong means the whole solution is wrong.
- **Determining preconditions.** A **precondition** is a condition that must already be true *before* a process can run correctly. Stating preconditions lets you assume them inside the routine rather than re-checking (e.g. a `binarySearch` precondition is "the list is already sorted").
- **Caching.** **Caching** means storing the results of expensive operations (or frequently-used data) somewhere fast so they can be reused without recomputing or re-fetching. *Benefits:* faster response, less repeated computation, less network/disk traffic. *Cost/trade-off:* uses extra storage, and the cache can become **stale** (out of date) if the underlying data changes — needing an invalidation strategy.
- **Reusable program components.** Writing self-contained, generalised modules (procedures, functions, classes, libraries) that can be **reused** in this and future projects. *Benefits:* less development time, fewer bugs (code is already tested), consistency, easier maintenance.

> **Worked scenario — a cinema booking website.**
> *Inputs:* film choice, date/time, number of seats, customer payment details.
> *Outputs:* a seating map, a booking confirmation, a receipt/e-ticket.
> *Precondition:* the chosen showing must exist and have enough free seats before the payment routine runs.
> *Caching:* the list of films and showtimes rarely changes during the day, so cache it on the web server rather than querying the database on every page load — far faster, but it must be refreshed when the schedule is updated (avoiding stale data).
> *Reusable components:* a `processPayment()` module and a `sendEmail()` module can be reused for tickets, refunds and the gift-shop checkout.

> 💡 **A\* extension.** Distinguish *temporal* caching (memoisation — storing computed results, e.g. cached Fibonacci values) from *spatial* caching (browser/CDN caching of static assets). Top answers discuss **cache invalidation** ("there are only two hard things in computer science…") and the storage-vs-speed trade-off explicitly, and link reusable components to lower **maintenance cost** and improved **reliability**, not just development speed.

### 2.1.3 Thinking procedurally

Thinking procedurally is breaking a problem into an ordered sequence of steps and sub-procedures.

- **Components of a problem.** Identify the separate sub-problems the overall problem is made of.
- **Components of a solution.** Identify the separate sub-tasks (modules/procedures) the solution will need.
- **Order of steps.** Determine the correct **sequence** — some steps depend on the output of earlier steps and cannot be reordered (you must take payment *before* issuing a ticket).
- **Decomposition.** **Decomposition** is breaking a large problem down into smaller, more manageable sub-problems, each solved by a **sub-procedure** (subroutine). This makes the problem easier to understand, lets a team divide work, and promotes reuse.

> **Worked scenario — making an automated cup of tea (a robot).** Decompose into sub-procedures: `boilWater()`, `addTeabag()`, `pourWater()`, `wait()`, `removeTeabag()`, `addMilk()`. The **order matters**: you cannot `pourWater()` before `boilWater()`, and `removeTeabag()` must follow `wait()`. Each sub-procedure could be developed and tested independently.

> 💡 **A\* extension.** Procedural thinking is about identifying **data dependencies** between steps (which steps *must* precede which) versus steps that are merely conventional. This directly feeds 2.1.5: any two sub-procedures with *no* dependency between them are candidates for concurrent execution. Strong answers map decomposition to a clear **hierarchy chart** and justify the chosen granularity.

### 2.1.4 Thinking logically

Thinking logically is identifying the points in a problem where the path of execution depends on a condition.

- **Decision points.** A **decision point** is a place in the solution where the flow of execution can branch in different directions depending on whether a condition is true or false (an `IF`, `CASE/SWITCH`, or loop test).
- **Structure determined by decisions.** The decisions determine the **flow of control / structure** of the program — which statements run, in which order, and how many times (selection and iteration are built from decision points).
- **Conditions at decision points.** For each decision point you must state the exact **condition** (the Boolean test) and what happens for each outcome. Getting the condition wrong (e.g. `>` instead of `>=`) changes the behaviour.

> **Worked scenario — an ATM withdrawal.**
> *Decision point 1:* `IF pin_correct` → continue, `ELSE` → re-prompt / retain card.
> *Decision point 2:* `IF amount <= balance AND amount <= dailyLimit` → dispense, `ELSE` → refuse.
> *Decision point 3:* `IF cashAvailable >= amount` → dispense, `ELSE` → show "machine empty".
> The whole structure of the transaction — and which messages appear — is determined by these three conditions.

> 💡 **A\* extension.** At the top band, examiners want **complete, correctly-bounded conditions**: combined Boolean logic (`AND`/`OR`), correct handling of edge/boundary cases (exactly the limit, zero, empty input), and identifying *implicit* decision points such as a loop's exit condition. Drawing a flowchart or decision table to show every path is a strong way to demonstrate this.

### 2.1.5 Thinking concurrently

Thinking concurrently is spotting which parts of a problem can be carried out **at the same time** (in parallel or interleaved) rather than one after another.

- **Determining concurrent processes.** Tasks that are **independent** — they do not depend on each other's output and do not compete for the same single resource — can run concurrently. Tasks with a data dependency must stay sequential.
- **Benefits.** Faster overall completion (more done per unit time / better throughput), better use of multi-core hardware, and a more responsive system (the UI stays usable while a long task runs in the background).
- **Trade-offs.** Concurrency is **harder to design, program and debug**; it can introduce **race conditions** and **deadlock**; results may need to be **synchronised/recombined**; and the management overhead can mean it is *not always faster* (especially on a single processor where tasks are only time-sliced, not truly parallel). Some problems are inherently sequential and gain nothing.

> **Worked scenario — rendering a weather-forecast website.** Independent tasks that can run concurrently: fetch the radar image, fetch the 5-day forecast data, fetch the news ticker, and load the advertising banner — none depends on another, so doing them in parallel makes the page appear faster. But *generating the final temperature graph* must wait until the forecast data has arrived, so that step stays sequential (a data dependency).

> 💡 **A\* extension.** Distinguish **true parallelism** (multiple cores running tasks simultaneously) from **concurrency via time-slicing** on a single core (the illusion of simultaneity). Discuss **Amdahl's Law** intuitively: speed-up is limited by the proportion of the task that *must* remain sequential, so adding cores gives diminishing returns. Reference race conditions/deadlock and the need for locks or other synchronisation as the price of safe concurrency.

## Worked examples

### Worked example A — Designing a computer chess game

| Thinking skill | Application to the chess game |
|---|---|
| **Abstract** | *Representational:* model the board as an 8×8 grid (2-D array); each piece as an object with a type, colour and position; ignore the physical look/weight of the pieces. *Generalisation:* every piece shares attributes (colour, position) and a `getLegalMoves()` method, so use a general `Piece` category specialised into `Pawn`, `Knight`, etc. The model differs from reality — it captures *rules and positions*, not wooden textures. |
| **Ahead** | *Inputs:* the player's chosen move (from-square, to-square). *Outputs:* updated board display, check/checkmate message. *Precondition:* it must be that player's turn and the move must be legal. *Caching:* store (memoise) the evaluation scores of board positions the AI has already analysed so it doesn't recompute them. *Reusable components:* `isMoveLegal()` is reused by both players and by the AI. |
| **Procedural** | Decompose into `validateMove()`, `updateBoard()`, `checkForCheck()`, `switchPlayer()`, `aiChooseMove()`. *Order:* validate → update → check for check/checkmate → switch turn. |
| **Logical** | Decision points: *is the move legal?*; *does this move leave the king in check (illegal)?*; *is the opponent now in checkmate / stalemate?*. Each condition (e.g. `IF kingHasNoLegalMoves AND kingInCheck` → checkmate) determines what happens next. |
| **Concurrent** | The AI can evaluate several candidate move-trees concurrently on different cores to choose its move faster; the UI animation can run concurrently with the AI "thinking" so the interface stays responsive. Trade-off: combining the parallel search results and avoiding shared-data races adds complexity. |

### Worked example B — Designing a hotel-room booking system

- **Abstract:** represent each room as a record (room number, type, price, availability calendar); ignore irrelevant detail like wallpaper colour. Generalise "single", "double" and "suite" into a `Room` category with a `pricePerNight` attribute.
- **Ahead:** *Inputs* = dates, room type, guest details, payment. *Outputs* = availability list, confirmation, invoice. *Precondition:* the room must be free for *every* night of the requested range before payment. *Caching:* cache the room-type/price list (changes rarely) for fast searching. *Reusable components:* a `checkAvailability(room, dateRange)` function reused on the search page and the confirmation page.
- **Procedural:** `searchRooms()` → `selectRoom()` → `checkAvailability()` → `takePayment()` → `confirmBooking()`. Order is fixed: never take payment before confirming availability.
- **Logical:** decision points — *are any rooms free?*, *is payment authorised?*, *is the stay over the minimum nights?* Condition example: `IF roomsFree > 0` show results `ELSE` show "no availability".
- **Concurrent:** search availability across *several hotels in a chain* concurrently; render the page banner and map while the search runs. Trade-off: must merge results and lock a room during booking to prevent two customers double-booking the same room (race condition).

## Key terms (exam definitions)

| Term | Definition |
|---|---|
| **Abstraction** | The process of removing/hiding unnecessary detail so only the information relevant to the problem remains. |
| **Representational abstraction** | Removing unnecessary detail from a *specific* thing to leave a simplified representation of that problem. |
| **Generalisation (categorisation) abstraction** | Grouping things by shared characteristics to form a more general, reusable model. |
| **Input** | Data that must be supplied *to* a system for it to produce a result. |
| **Output** | The result(s) a system must produce. |
| **Precondition** | A condition that must be true *before* a process can run correctly. |
| **Caching** | Storing results of expensive/frequent operations in fast storage so they can be reused without recomputing or re-fetching. |
| **Stale data** | Cached data that is out of date because the underlying source has changed. |
| **Reusable component** | A self-contained, generalised module that can be used in this and other programs. |
| **Decomposition** | Breaking a large problem into smaller, more manageable sub-problems. |
| **Sub-procedure (subroutine)** | A named block of code that solves one sub-problem, called by the main program. |
| **Decision point** | A point where the flow of execution branches depending on a condition. |
| **Flow of control** | The order in which a program's statements are executed, shaped by its decision points. |
| **Concurrency** | Carrying out (or interleaving) more than one task over the same period of time. |
| **Parallel processing** | Executing multiple tasks literally simultaneously on multiple processors/cores. |
| **Race condition** | A fault where the result depends on the unpredictable timing of concurrent tasks accessing shared data. |
| **Deadlock** | A standstill where concurrent tasks each wait for a resource the other holds, so none can proceed. |

## Common exam mistakes & examiner tips

- **Defining instead of applying.** The biggest lost-marks trap. If the question gives a scenario, *every* point must reference that scenario (name the actual inputs, the actual decision points). A generic "abstraction removes detail" with no link to the scenario scores poorly on AO2/AO3.
- **Confusing the two types of abstraction.** *Representational* = simplify one specific thing; *generalisation* = group many things by shared features. Give a scenario-specific example of each.
- **Abstraction vs decomposition.** Abstraction *removes/hides* detail; decomposition *breaks a problem into parts*. They are different skills — don't blur them.
- **Inputs/outputs muddled with processing.** "Validate the data" is a *process*, not an input or output. Inputs are data in; outputs are results out.
- **Caching answers that ignore the downside.** Always pair the benefit (speed) with a trade-off (extra storage / stale data) when asked to "discuss" or "evaluate".
- **Vague conditions.** "If there are enough seats" is weaker than "`IF seatsRequested <= seatsAvailable`". State the actual Boolean condition and the outcome of *both* branches.
- **Claiming concurrency for dependent tasks.** You cannot do two tasks at once if one needs the other's output. Always check for a data dependency before saying tasks are concurrent.
- **Assuming concurrency is always faster.** On a single core it is time-sliced, not truly parallel, and management overhead can make it slower. Mention this when evaluating.
- **Watch the command word.** *State/identify* (AO1) wants brief facts; *describe/explain* (AO2) wants applied reasoning; *discuss/evaluate/justify* (AO3) wants pros, cons and a judgement.

## Exam-style questions

**Q1 (AO1, 2 marks).** State what is meant by *abstraction* and give **one** reason it is needed when solving a problem.
<details><summary>Mark scheme</summary>

- (1) Removing / hiding unnecessary detail so only relevant information remains.
- (1) Any one reason: makes the problem simpler/manageable to solve; reduces complexity; lets us focus on what is relevant; makes a model reusable/understandable.
</details>

**Q2 (AO2, 4 marks).** A company is writing a sat-nav app. Using a road network as your example, explain the difference between **representational abstraction** and **generalisation/categorisation abstraction**.
<details><summary>Mark scheme</summary>

*Representational (up to 2):*
- Removing unnecessary detail from the specific road network (1) e.g. modelling it as a graph of nodes (junctions) and weighted edges (road distances), ignoring scenery/road surface/traffic lights (1).

*Generalisation (up to 2):*
- Grouping things by shared characteristics (1) e.g. treating all roads as a general "edge with a length/speed", or all junctions as a general "node", so the same routing algorithm works for any road (1).
- Max 4. Must be applied to the road/sat-nav scenario.
</details>

**Q3 (AO2, 5 marks).** A school is building an online homework-submission system. Identify **two inputs** and **two outputs**, state **one precondition** that must hold before a submission is accepted, and explain how **caching** could improve performance.
<details><summary>Mark scheme</summary>

- Inputs (1 each, max 2): student ID/login, the homework file, the assignment chosen, the submission timestamp.
- Outputs (1 each, max 2): confirmation message/receipt, updated submitted-status, late flag, teacher notification.
- Precondition (1): student must be logged in / the assignment must exist / the deadline check requires the current date — i.e. one valid condition that must be true *before* accepting.
- Caching (1): store frequently-read data such as the class/assignment list so it isn't re-fetched from the database on every page load → faster response (must explain *what* is cached and the benefit).
- Max 5.
</details>

**Q4 (AO2, 4 marks).** A program plays a vending machine. Describe the **decision points** involved and the **conditions** that determine the outcome at each.
<details><summary>Mark scheme</summary>

Award 1 per correctly described decision point + condition, max 4. Examples:
- *Is the product in stock?* `IF stock[item] > 0` → continue, else "sold out".
- *Has enough money been inserted?* `IF moneyInserted >= price` → continue, else wait for more.
- *Is change available?* `IF changeDue <= coinsInMachine` → dispense + change, else refuse/return money.
- *Dispense:* outcome determined by all above being true.
Conditions must be specific (a Boolean test), not just naming the step.
</details>

**Q5 (AO3, 6 marks).** A video-streaming service must, for each user, (a) fetch the recommended-films list, (b) fetch the "continue watching" list, (c) load the homepage banner advert, and (d) start buffering the most-recently-watched film. Discuss which of these tasks could be performed **concurrently**, and evaluate the **benefits and trade-offs** of doing so.
<details><summary>Mark scheme</summary>

Indicative content — credit a reasoned discussion:
- Tasks (a), (b), (c), (d) are largely independent (no task needs another's output) → all are candidates for concurrent execution (1–2).
- Benefit: the homepage loads much faster / appears responsive because fetches happen in parallel rather than waiting one-after-another; better use of multi-core hardware and network (1–2).
- Trade-off: harder to program/debug; possible contention for limited network bandwidth so not linear speed-up; results must be combined when all arrive; on a single core it is only time-sliced (1–2).
- Note a dependency if present (e.g. rendering the combined homepage must wait for all fetches) (1).
- Judgement: concurrency is worthwhile here because the tasks are independent and I/O-bound, but the developer must manage synchronisation (1).
Bands: 5–6 = sustained, balanced discussion applied to the scenario with a justified conclusion; 3–4 = some application + some trade-offs; 1–2 = generic points.
</details>

**Q6 (AO2/AO3, 6 marks).** A games company is designing a multiplayer racing game. Explain how **decomposition** and **reusable program components** would be used in its development, applying both to the racing-game scenario.
<details><summary>Mark scheme</summary>

*Decomposition (up to 3):* break the game into sub-problems each with a sub-procedure, e.g. `handlePlayerInput()`, `updatePhysics()`, `detectCollisions()`, `renderFrame()`, `updateLeaderboard()`, `networkSync()`; state that smaller parts are easier to develop/test and can be split across a team. Must name scenario-relevant sub-procedures.

*Reusable components (up to 3):* identify generalised modules reused across the game/future games, e.g. a `Vector2D`/physics module reused by every car, a `collision` routine reused for cars, walls and pickups, a `network` library reused across game modes; benefits = faster development, fewer bugs (already tested), consistency, easier maintenance.
- Max 6; must be applied to the racing game.
</details>

**Q7 (AO3, 8 marks).** "Building an abstract model of a problem always makes the final software better." Discuss, with reference to the **differences between abstraction and reality**, using a flight-simulator (for training pilots) as your example.
<details><summary>Mark scheme</summary>

Indicative content (balanced discussion + judgement required for top band):
- *For:* abstraction removes irrelevant detail (exact rivet positions, cabin upholstery) so designers focus on flight dynamics → manageable, faster to build, cheaper, reusable across aircraft types.
- *Against / limits:* an abstraction is a simplification, so it *differs from reality* — omitting crosswind, turbulence or instrument failure would make the simulator unrealistic and dangerous for training; over-abstraction loses the fidelity that the problem actually requires.
- *Key point:* the value depends on choosing the *right* level of abstraction — discard only detail irrelevant to *this* purpose (training pilots).
- Judgement: abstraction is essential but not automatically beneficial; a poorly chosen abstraction can make software worse.
- Bands: 7–8 sustained two-sided argument applied to the scenario with justified conclusion; 5–6 both sides, some application; 3–4 mostly one-sided; 1–2 generic definitions.
</details>
